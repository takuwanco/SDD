## 仕様駆動開発の4つの原則

仕様駆動開発は、以下の4つの原則に基づいて実践されます：

### ① 仕様は"生きたドキュメント"

**原則の内容**
仕様はプロジェクトとともに進化する、継続的に更新される動的な成果物である。必要に応じて改良・細分化し、常に最新のプロジェクト状況を反映する。

**実践のポイント**
- 仕様をMarkdownやREADMEといった、テキストベースの形式で記述する
- Gitなどのバージョン管理システムで仕様を管理し、すべての変更履歴を残す
- 仕様の更新を「特別な作業」ではなく、日常的な開発活動の一部とする
- AIを活用して、仕様の更新コストを削減する

**陥りやすい誤解**
「生きたドキュメント」は「いつでも自由に変えていい」という意味ではありません。変更は必ず履歴として残し、関係者に共有され、影響範囲が分析されます。

### ② 仕様は"信頼できる唯一の情報源"

**原則の内容**
会話やメール、散逸した資料に頼らず、すべてのメンバーが「仕様」に立ち返る。迷ったときは仕様に戻り、そこを出発点に修正や意思決定を行う。

**実践のポイント**
- 重要な意思決定は、必ず仕様に反映する（口頭やメールだけで終わらせない）
- 仕様へのアクセス権を、チーム全員（可能なら顧客も）に開放する
- 議論が平行線になったとき、「仕様には何と書いてあるか」を確認する習慣をつける
- 新メンバーのオンボーディングは、仕様を読むことから始める

**陥りやすい誤解**
「信頼できる唯一の情報源」は「すべてを仕様に書く」という意味ではありません。会話やメールは重要です。しかし、**最終的な合意事項や意思決定は仕様に反映する**ことで、「信頼できる唯一の情報源」として機能させるのです。

### ③ 仕様は"変更と反復が前提"

**原則の内容**
プロジェクトが複雑化したら仕様を改善する。変化を例外扱いするのではなく、必然として取り込み続ける。

**実践のポイント**
- 契約段階で「仕様は進化するもの」と明記し、変更を前提とした契約形態を選択する
- 仕様の変更履歴を差分レポートとして可視化し、「何が、なぜ変わったか」を透明にする
- 反復（iteration）のサイクルを明確にし、各サイクルで仕様を見直す機会を設ける
- 変更を「失敗」ではなく「学びの反映」として肯定的に捉える文化を育てる

**陥りやすい誤解**
「変更と反復が前提」は「計画不要」という意味ではありません。むしろ、変更が起きることを前提に計画するからこそ、より現実的で柔軟な計画が立てられるのです。

### ④ AIでコストを抑える

**原則の内容**
AIが自然言語を理解できるようになったからこそ、上記3つの原則（生きたドキュメント、信頼できる唯一の情報源、変更と反復）を実現できる。仕様を唯一の入力として、AIが計画・タスク・実装の支援を行い、複数の人と複数のAIが同じ参照点を共有する。

**実践のポイント**
- 仕様をAIが理解できる形式（自然言語のMarkdown、構造化されたOpenAPIなど）で記述する
- AIに設計案やコードを生成させる際、常に仕様を入力として与える
- AIの出力を鵜呑みにせず、人間がレビューする「ヒューマン・イン・ザ・ループ」を徹底する
- AIを「ペアプログラマ」「アシスタント」として位置づけ、最終判断は人間が行う

**陥りやすい誤解**
「AIでコストを抑える」は「AIに丸投げする」という意味ではありません。AIは強力なツールですが、**「何を作るか」を定義するのは人間の責任**です。AIは、その実現を支援するパートナーなのです。

### 4つの原則の相互関係

これら4つの原則は、独立しているのではなく、相互に支え合っています。

- 原則①（生きたドキュメント）が、原則②（信頼できる唯一の情報源）として機能する基盤を作る
- 原則②（信頼できる唯一の情報源）が、原則③（変更と反復）を安全に行える環境を提供する
- 原則③（変更と反復）が、原則①（生きたドキュメント）の必要性を生み出す
- 原則④（AI活用）が、原則①（生きたドキュメント）の実現コストを下げる

## 仕様駆動開発の7つの工程

仕様駆動開発は、以下の7つの工程で実践されます。各工程ごとに1つのMarkdownファイルで管理します。

### ① 原則決定工程

**目的**: プロジェクトの存在意義・原則・制約条件を明文化し、合意形成する

**成果物**: [`01-principle-definition.md`](./01-principle-definition.md)（プロジェクト憲章）

**特徴**:
- PMBOKのProject Charterに相当するが、Spec-as-Codeの思想に基づき、機械可読な文書として管理
- プロジェクトの価値観・責任範囲・ガバナンスをコード同様に差分・履歴管理

**ポイント**: プロジェクト開始時に必ず策定し、仕様・設計・実装すべての「前提」として扱う

### ② 企画・要件定義工程

**目的**: 顧客やステークホルダーと共に「何を」「なぜ」作るのかを定義する

**成果物**: [`02-planning-requirement.md`](./02-planning-requirement.md)（仕様書）

**特徴**:
- 従来の要件定義書に相当するが、変更を前提に「進化する仕様」として扱う
- ユーザー体験やビジネスゴールを中心に仕様をまとめ、AIに詳細化を支援させる

**ポイント**: 契約段階で「仕様は変わる」ことを顧客に合意してもらい、履歴管理や差分レポートを導入

### ③ 設計計画工程

**目的**: 技術スタック、アーキテクチャ、制約条件を整理し、AIを活用して設計プランを策定する

**成果物**: [`03-design-planning.md`](./03-design-planning.md)（設計計画書）

**特徴**:
- 複数案をAIから引き出し、チームで比較検討する
- 日本の現場で重視される「合意形成」に資するドキュメントとして残す

**ポイント**: レビュー会を設け、AI提案をそのまま採用せず、ヒューマン・イン・ザ・ループを徹底

### ④ タスク分割工程

**目的**: 仕様と設計をもとに、レビューやテスト可能な粒度でタスクを分割する

**成果物**: [`04-task-breakdown.md`](./04-task-breakdown.md)（タスク分割書）

**特徴**:
- 「1タスク＝レビューが30分以内に終わる大きさ」などのガイドラインを設ける
- 依存関係と優先度を明示化し、自動化ツールで進捗を可視化

**ポイント**: 日本式の進捗報告会に流用できるよう、バーンダウンチャートやガントチャートを出力

### ⑤ 実装工程

**目的**: タスクごとにCursorなどのAIエディタにコードを書かせ、人間がレビュー・検証する

**成果物**: [`05-implementation.md`](./05-implementation.md)（実装記録）

**特徴**:
- 小さな変更単位でコミットし、レビュー負荷を軽減
- テストコードもAIに生成させ、CI/CDで自動検証

**ポイント**: 「人が必ず確認する」ことを工程に組み込み、AI出力を鵜呑みにしない（ヒューマン・イン・ザ・ループ）

### ⑥ 検証・受入工程

**目的**: 全体の仕様が満たされているかを顧客と共に確認する

**成果物**: [`06-verification-acceptance.md`](./06-verification-acceptance.md)（検証記録）

**特徴**:
- 従来の「総合テスト工程」にあたるが、仕様差分レポートを活用し、変更履歴を顧客に説明
- 仕様更新と実装の整合性をトレースできる状態で提示

**ポイント**: 公的機関や大企業顧客でも納得できるように、文書化と透明性を強化

### ⑦ 移行・運用工程

**目的**: システムを本番へ移行し、運用フェーズで仕様変更や改善を継続的に反映する

**成果物**: [`07-migration-operation.md`](./07-migration-operation.md)（運用記録）

**特徴**:
- 運用中のフィードバックを次の「仕様定義」に戻すサイクルを明示化
- ウォーターフォールの「一方向性」を残しつつ、循環性を取り入れる

**ポイント**: 日本の「保守契約」文化に合わせ、仕様駆動開発の反復を運用改善サイクルに組み込む

## 使い方

### 1. 各工程のファイルを参考にする

実際のプロジェクトで仕様駆動開発を実践する際は、各工程のサンプルファイルを参考にしてください。

- プロジェクト開始時：[`01-principle-definition.md`](./01-principle-definition.md)を参考にプロジェクト憲章を作成
- 要件定義時：[`02-planning-requirement.md`](./02-planning-requirement.md)を参考に仕様書を作成
- 設計時：[`03-design-planning.md`](./03-design-planning.md)を参考に設計計画書を作成
- 以降も同様に、各工程のサンプルファイルを参考に進める

### 2. ファイル名の命名規則

実際のプロジェクトでは、以下のような命名規則を推奨します：

- 原則決定工程：`Constitution.md` または `01-principle-definition.md`
- 企画・要件定義工程：`spec.md` または `02-planning-requirement.md`
- 設計計画工程：`design.md` または `03-design-planning.md`
- タスク分割工程：`tasks.md` または `04-task-breakdown.md`
- 実装工程：`implementation.md` または `05-implementation.md`
- 検証・受入工程：`verification.md` または `06-verification-acceptance.md`
- 移行・運用工程：`operation.md` または `07-migration-operation.md`

### 3. ファイル間の参照

各ファイルは、前の工程の成果物を参照します：

- [`02-planning-requirement.md`](./02-planning-requirement.md) → [`01-principle-definition.md`](./01-principle-definition.md)を参照
- [`03-design-planning.md`](./03-design-planning.md) → [`02-planning-requirement.md`](./02-planning-requirement.md)を参照
- [`04-task-breakdown.md`](./04-task-breakdown.md) → [`03-design-planning.md`](./03-design-planning.md)と[`02-planning-requirement.md`](./02-planning-requirement.md)を参照
- 以降も同様に、前の工程の成果物を参照

### 4. 4つの原則に基づく運用

各ファイルを更新する際は、4つの原則を意識してください：

- **原則①（生きたドキュメント）**：必要に応じて更新し、変更履歴を残す
- **原則②（信頼できる唯一の情報源）**：重要な意思決定は必ずファイルに反映する
- **原則③（変更と反復が前提）**：変更を恐れず、履歴として記録する
- **原則④（AIでコストを抑える）**：AIを活用して更新作業を支援する

## サンプルプロジェクト：顧客管理システム

このディレクトリのサンプルファイルは、すべて「顧客管理システム」という仮想のプロジェクトを例として作成されています。

### プロジェクトの概要

- **目的**: 営業担当者が増えたため、Excelでの顧客管理が限界に達している。リアルタイムで情報を共有できるシステムが必要
- **想定ユーザー**: 営業担当者、営業マネージャー、経営層
- **主要機能**: 顧客情報の管理、営業活動履歴の記録、レポート・分析機能、権限管理

### 各工程の例

各ファイルには、このプロジェクトの各工程での成果物の例が記載されています。実際のプロジェクトでも、同様の構成でファイルを作成してください。

## ライセンスと注意事項

### ライセンス

このサンプルファイルは、本書の読者が練習のために使用することを目的としています。自由に編集・利用してください。

### 注意事項

1. **このサンプルは参考用です**
   - 実際のプロジェクトでは、プロジェクトの特性に合わせて内容を調整してください
   - すべての項目を必ず記載する必要はありません

2. **機密情報の取り扱い**
   - 実際のプロジェクトで使用する場合は、機密情報が含まれないように注意してください
   - 必要に応じて、`.gitignore`で機密情報を含むファイルを除外してください

3. **ファイルの更新**
   - 仕様駆動開発の原則①「生きたドキュメント」に基づき、必要に応じてファイルを更新してください
   - 変更履歴はGitで管理されます

4. **AIの活用**
   - 原則④「AIでコストを抑える」に基づき、AIを活用してファイルの作成や更新を支援してください
   - ただし、AIの出力は必ず人間がレビューし、承認してください（ヒューマン・イン・ザ・ループ）

5. **チームでの共有**
   - 原則②「信頼できる唯一の情報源」に基づき、チーム全員がこれらのファイルを参照できるようにしてください
   - GitHubなどの共有リポジトリで管理することを推奨します
