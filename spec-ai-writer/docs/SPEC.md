# Spec AIライター 仕様書

## 1. 概要

### 1.1 システム概要

Spec AIライター は、LLM（大規模言語モデル）を活用した対話形式のインタビューシステムである。ユーザーとの対話を通じて情報を収集し、仕様駆動開発（Specification Driven Development）で定義された7つの仕様書を自動生成する。

### 1.2 目的

- プロジェクト開始時の仕様策定プロセスを効率化
- LLMによる自然言語対話で非技術者でも仕様書作成が可能
- 7つの工程の標準化されたワークフローにより品質を担保
- 生成された仕様書のGit管理による変更履歴の追跡

### 1.3 対象ユーザー

- プロジェクトマネージャー
- プロダクトオーナー
- エンジニアリングリード
- 仕様策定に関わるステークホルダー

## 3. 機能要件

### 3.1 7つの工程 インタビュー機能

システムは以下の7つの工程を順番に実行し、各工程で仕様書を生成する。

| 工程 | 名称 | 生成ファイル | 収集情報 |
|------|------|-------------|----------|
| 1 | 原則決定工程 | 01-principle-definition.md | 背景、目的、原則、スコープ、制約、ステークホルダー、成功基準 |
| 2 | 企画・要件定義工程 | 02-planning-requirement.md | 概要、対象ユーザー、機能要件、非機能要件 |
| 3 | 設計計画工程 | 03-design-planning.md | 技術選定、アーキテクチャ、設計判断 |
| 4 | タスク分割工程 | 04-task-breakdown.md | マイルストーン、タスク、依存関係、優先度 |
| 5 | 実装工程 | 05-implementation.md | 実装記録、レビュー結果、AI利用記録 |
| 6 | 検証・受入工程 | 06-verification-acceptance.md | テスト項目、結果、仕様差分、受入結果 |
| 7 | 移行・運用工程 | 07-migration-operation.md | 移行計画、運用体制、フィードバック、メトリクス |

### 3.2 工程依存関係

```
工程1（原則決定工程）
    ↓
工程2（企画・要件定義工程）← 工程1の情報を参照
    ↓
工程3（設計計画工程）← 工程1,2の情報を参照
    ↓
工程4（タスク分割工程）← 工程2,3の情報を参照
    ↓
工程5（実装工程）← 工程2,3,4の情報を参照
    ↓
工程6（検証・受入工程）← 工程2,5の情報を参照
    ↓
工程7（移行・運用工程）← 工程6の情報を参照
```

### 3.3 インタビュー処理仕様

#### 質問生成ルール
- LLMが工程のシステムプロンプトに基づき質問を生成
- 1回につき1つの質問を提示
- ユーザー回答に基づきフォローアップ質問を動的生成
- 不明確な回答には具体例を示して再質問

#### 工程完了条件
- 最低3回以上のQ&Aが完了
- 必須フィールドがすべて収集済み（スキーマ検証成功）
- 最大15回のQ&Aで強制終了（安全上限）

#### 中断・再開機能
- Ctrl+C または "exit" 入力で中断
- 進捗は自動的にJSONファイルに保存
- `spec resume <project>` で再開可能

### 3.4 仕様書生成機能

- Jinja2テンプレートによるMarkdown生成
- プロジェクト名、生成日時を自動挿入
- 変更履歴テーブルを自動追加
- 出力先: `./spec_output/{project_name}/` ディレクトリ（プロジェクトごとに分離）

### 3.5 Git連携機能

- `AUTO_GIT_COMMIT=true` で有効化
- 各工程完了時に自動コミット
- コミットメッセージ形式: 「工程N: 仕様書生成」

### 3.6 マルチプロバイダー対応

| プロバイダー | デフォルトモデル | 用途 |
|-------------|----------------|------|
| Claude (Anthropic) | claude-3-5-sonnet-20241022 | 汎用（推奨） |
| OpenAI | gpt-4-turbo-preview | 代替 |
| AWS Bedrock | claude-3-5-sonnet (リージョン指定) | エンタープライズ |

### 3.7 LLMクライアント インターフェース仕様

#### 3.7.1 クラス構成

```
BaseLLMClient (抽象基底クラス)
    ├── ClaudeClient      # Anthropic API
    ├── OpenAIClient      # OpenAI API
    └── BedrockClient     # AWS Bedrock API
```

#### 3.7.2 BaseLLMClient インターフェース

すべてのLLMクライアントは `BaseLLMClient` を継承し、以下のメソッドを実装する。

| メソッド | 引数 | 戻り値 | 説明 |
|---------|------|--------|------|
| `chat()` | `messages: List[Dict[str, str]]`, `temperature?`, `max_tokens?` | `str` | チャットメッセージを送信し応答を取得 |
| `generate_question()` | `system_prompt: str`, `context: Dict[str, Any]` | `str` | インタビュー質問を生成 |
| `extract_structured_data()` | `conversation: str`, `schema: Dict[str, Any]` | `Dict[str, Any]` | 会話から構造化データを抽出 |

#### 3.7.3 メッセージ形式

`chat()` メソッドに渡すメッセージは以下の形式とする。

```
[
  {"role": "system", "content": "システムプロンプト"},
  {"role": "user", "content": "ユーザーメッセージ"},
  {"role": "assistant", "content": "アシスタント応答"}
]
```

各クライアントは内部でAPIの仕様に応じて変換を行う。
- **OpenAI**: そのまま使用
- **Anthropic**: `system` を別パラメータに分離
- **Bedrock**: `system` を別フィールドに分離、`content` をリスト形式に変換

#### 3.7.4 共通プロンプト定義

プロバイダー間で一貫した動作を保証するため、以下のプロンプトは `BaseLLMClient` で定義し、全クライアントで共有する。

**質問生成時の指示文:**
```
次の質問を1つだけ生成してください。質問のみを出力し、説明や前置きは不要です。
```

**構造化データ抽出用システムプロンプト:**
```
あなたは会話から構造化データを抽出する専門家です。
与えられた会話から、指定されたスキーマに従ってデータを抽出し、JSON形式で返してください。

重要な指示:
1. 会話に明示的に含まれていない情報は推測しないでください
2. リストや配列が適切な場合は配列として返してください
3. 情報が欠けている場合は null を使用してください
4. JSON形式で返してください（マークダウンのコードブロックは不要）
```

**構造化データ抽出用ユーザープロンプトテンプレート:**
```
以下の会話から、次のスキーマに従ってデータを抽出してください:

{schema_description}

会話:
{conversation}

JSON形式で抽出したデータを返してください:
```

#### 3.7.5 コンテキスト構築

`generate_question()` で使用するコンテキストは `_build_context_prompt()` メソッドで構築する。

**入力 context の構造:**
```
{
  "conversation_history": "質問1: ...\n回答1: ...",
  "missing_fields": ["field1", "field2"],
  "previous_phases": { ... }
}
```

**構築されるプロンプト:**
```
{system_prompt}

## 会話履歴:
{conversation_history}

## まだ収集が必要な情報:
- field1
- field2

## 前工程の情報:
{previous_phases}
```

## 6. データモデル

### 6.1 プロジェクト

| フィールド | 型 | 説明 |
|-----------|-----|------|
| name | string | プロジェクト識別子（一意） |
| description | string? | プロジェクト説明 |
| current_phase | int (1-7) | 現在のフェーズ番号 |

| total_qa_pairs | int | 累計Q&A数 |

## 10. 制約事項

### 10.1 パフォーマンス制約

| 項目 | 制約値 |
|-----|--------|
| 最大Q&A回数（1フェーズ） | 15回 |
| LLMタイムアウト | 30秒 |
| 最大トークン数（生成） | 4,096 |
| WebSocket再接続試行 | 最大5回 |

### 10.2 動作環境

| 項目 | 要件 |
|-----|------|
| Python | 3.9 以上 |
| Node.js | 18 以上（フロントエンド） |
| ブラウザ | Chrome, Firefox, Safari（最新版） |

### 10.3 制限事項

- 同時に複数のインタビューセッションは非対応
- オフラインでのLLM利用は不可
- 生成仕様書の自動翻訳は非対応
