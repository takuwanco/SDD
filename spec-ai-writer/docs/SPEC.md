# 企画・要件定義工程

このファイルは、仕様駆動開発の**企画・要件定義工程**で作成される仕様書です。

## 仕様駆動開発の4つの原則に基づく運用

この仕様書は、以下の4つの原則に基づいて管理されます：

1. **仕様は"生きたドキュメント"**：プロジェクトとともに進化します
2. **仕様は"共通の信頼できる唯一の情報源"**：すべてのメンバーがこの仕様を参照します
3. **仕様は"変更と反復が前提"**：変更履歴を記録しながら更新します
4. **AIでコストを抑えて実現する**：AIを活用して仕様の詳細化やレビューを行います

## プロジェクト概要

### 目的

AIとの対話形式のインタビューにより、SDD仕様書の作成を自動化する。
SDD方法論の実践ハードルを下げ、誰でも一定品質の仕様書を作成できるようにする。

詳細は、[CONSTITUTION.md](./CONSTITUTION.md)（プロジェクト憲章）を参照してください。

### 背景

- SDD仕様書は7工程分あり、手作業で書くには時間がかかりすぎる
- 各工程で何をどこまで書けばよいか、経験者でないと判断できない
- SDDの方法論は優れているが、実践のハードルが高く導入が進まない
- LLMの対話能力を活用すれば、ヒアリングと文書化を同時に効率化できる

## 想定ユーザー

- **プロジェクトマネージャー**：プロジェクト開始時に仕様書を効率的に作成したい
- **プロダクトオーナー**：技術的な知識がなくても、対話形式で要件を整理したい
- **エンジニアリングリード**：設計判断やタスク分割の仕様書を素早く作成したい
- **SDD初学者**：方法論を学びながら、ガイド付きで仕様書を作成したい

## 機能要件

### 1. プロジェクト管理機能

- プロジェクトの新規作成・一覧表示・詳細確認・削除を行う
- プロジェクト作成時にシステムがユニークなプロジェクトID（16進数8文字、形式: `^[a-f0-9]{8}$`）を自動採番する
- プロジェクト作成時にユーザーが表示名と説明を設定する
- プロジェクトの表示名・説明は作成後に編集できる（`PATCH /api/projects/{project_id}`）
- プロジェクトに関するすべてのデータは、プロジェクトIDごとのディレクトリに集約する
- データディレクトリの構造は以下の通り：

```
{DATA_DIR}/                              # デフォルト: ./data
└── {project_id}/
    ├── project.json                     # メタ情報（ID、表示名、説明、作成日時、更新日時）
    ├── interview.json                   # インタビュー状態（Q&A履歴、工程進捗、構造化データ）
    └── specs/                           # 生成された仕様書
        ├── 01-principle-definition.md
        ├── 02-planning-requirement.md
        └── ...
```

### 2. 7工程インタビュー機能

- 7つの工程を順番にインタビュー形式で実施する
- 工程ごとのシステムプロンプトに基づき、LLMが質問を1つずつ生成する
- ユーザーの回答に基づいてフォローアップ質問を動的に生成する
- 不明確な回答には具体例を示して再質問する
- ユーザーからの質問や確認には先に回答してからインタビューを継続する
- 前工程で決定した内容について質問された場合は、前工程の仕様書を参照して回答する

| 工程 | 名称 | 生成ファイル | 収集情報 |
|------|------|-------------|----------|
| 1 | 原則決定工程 | 01-principle-definition.md | 背景、目的、原則、スコープ、制約、ステークホルダー、成功基準 |
| 2 | 企画・要件定義工程 | 02-planning-requirement.md | 概要、対象ユーザー、機能要件、非機能要件 |
| 3 | 設計計画工程 | 03-design-planning.md | 技術選定、アーキテクチャ、設計判断 |
| 4 | タスク分割工程 | 04-task-breakdown.md | マイルストーン、タスク、依存関係、優先度 |
| 5 | 実装工程 | 05-implementation.md | 実装記録、レビュー結果、AI利用記録 |
| 6 | 検証・受入工程 | 06-verification-acceptance.md | テスト項目、結果、仕様差分、受入結果 |
| 7 | 移行・運用工程 | 07-migration-operation.md | 移行計画、運用体制、フィードバック、メトリクス |

### 3. 工程完了判定機能

- 最低3回以上のQ&Aが完了していることを確認する
- LLMで会話履歴から構造化データ（JSON）を抽出する
- 構造化データの抽出に失敗した場合は1回リトライし、それでも失敗した場合はユーザーに警告する
- 工程ごとの必須フィールドがすべて収集済みであることをスキーマ検証する
- 最大15回のQ&Aで強制終了する

### 4. 仕様書自動生成機能

- 工程完了時にJinja2テンプレートからMarkdown仕様書を生成する
- プロジェクト名、生成日時、変更履歴テーブルを自動挿入する
- 出力先はプロジェクトディレクトリ内の `specs/` サブディレクトリ

### 5. 中断・再開機能

- インタビューの状態をプロジェクトディレクトリ内の `interview.json` に永続化する
- Q&A履歴、工程進捗、構造化データを保存する
- 中断後にプロジェクトIDを指定して同じ地点から再開できる
- 完了済みの工程をリセットして再インタビューできる（`POST /api/interview/reset-phase`）
- リセット時は対象工程のQ&A履歴・構造化データ・生成済み仕様書を削除する

### 6. マルチLLMプロバイダー対応

- 以下の3つのLLMプロバイダーに対応する
  - **AWS Bedrock**：推奨（Claude Haiku 4.5）
  - **Claude（Anthropic API）**：代替
  - **OpenAI API**：代替
- `.env` の `DEFAULT_LLM_PROVIDER` で切り替える
- 共通の `BaseLLMClient` インターフェースでプロバイダー差異を吸収する
- プロバイダー固有のエラーを共通例外クラス（`LLMAuthenticationError`, `LLMConnectionError`, `LLMResponseError`）に変換し、呼び出し元でプロバイダーに依存しないエラー処理を行う

### 7. Git自動コミット機能

- `AUTO_GIT_COMMIT=true` で有効化する
- プロジェクトごとに `specs/` ディレクトリをgitリポジトリとして管理する
- プロジェクト作成時に `specs/` ディレクトリで `git init` を実行する
- 各工程の仕様書生成後に自動コミットする

### 8. CLIインターフェース

- `spec start` で新規プロジェクト作成とインタビュー開始（プロジェクト名を対話で入力）
- `spec resume <project_id>` で中断したインタビュー再開
- `spec list` でプロジェクト一覧表示（ID・表示名・進捗）
- `spec status <project_id>` でプロジェクト進捗表示

### 9. Web UIインターフェース

- フロントエンドからバックエンドへの通信はREST API（Fetch API）で行う
- ダッシュボード画面でプロジェクトの一覧表示・新規作成・削除を行う
- チャット形式のインタビュー画面でLLMと対話する
- 工程進捗インジケーター（7段階）を表示する
- 生成済み仕様書のプレビュー・ダウンロード（個別/ZIP一括）を行う
- 仕様書画面から工程ごとの再インタビューを実行できる
- モックモード（`VITE_USE_MOCK_API=true`）でAPIキー不要のUI確認が可能
- `APP_ENV=production` でバックエンドがビルド済み静的ファイル（`frontend/dist`）を直接サーブする。`APP_ENV=development`（デフォルト）ではVite開発サーバーを使用する

## 非機能要件

### パフォーマンス

- LLMの質問生成は30秒以内にタイムアウトする
- 1工程あたりの最大Q&A回数は15回（無限ループ防止）
- LLMの最大トークン数は4,096（1回の生成あたり）

### 可用性

- インタビュー状態はQ&Aのたびにディスクに自動保存する
- プロセスが異常終了しても、直前のQ&Aまで復元できる

### 技術スタック

- バックエンド：Python 3.9以上 / FastAPI / uv（パッケージ管理）
- フロントエンド：Node.js 18以上 / Vite + React
- ブラウザ：Chrome、Firefox、Safari（最新版）

### セキュリティ

- LLM APIキーは `.env` ファイルで管理し、リポジトリにはコミットしない
- AWS認証情報は環境変数またはIAMロールで管理する
- サーバーは `127.0.0.1`（localhost のみ）でリッスンし、LAN 上の他ホストからのアクセスを遮断する
