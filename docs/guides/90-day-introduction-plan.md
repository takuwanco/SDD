# 90日間導入プラン

このドキュメントは、『仕様駆動開発 実践入門』で紹介されている仕様駆動開発を、90日間でチームに定着させるための実践的なプランです。

## このプランの目的

90日間で、仕様駆動開発がチームの「当たり前」になっている状態を目指します。完璧を求めず、段階的に実践を積み重ねていくことが重要です。

## 第1週：準備と小さく始める

最初の1週間は、完璧を求めず、できることから始めることが大切です。第5章「最初の1週間で仕様駆動開発を始める方法」を参照しながら、以下のタスクに取り組みましょう。

- [ ] チームメンバーの合意を得る
- [ ] GitHubリポジトリを作成
- [ ] README.mdに最初の仕様を書く（プロジェクト名と背景から始める）
- [ ] 最初のPull Requestを作成・マージ

**目標**: 仕様駆動開発の基本的な流れを体験する

## 第2〜4週：習慣化

最初の1週間を乗り切ったら、次は習慣化を目指します。毎日少しずつでも仕様を更新し、Pull Requestを通じて変更を共有する習慣を作りましょう。

- [ ] 週3件以上のPull Requestを目標に
- [ ] 全員が仕様レビューに参加
- [ ] 小さな成功体験を共有
- [ ] トラブルシューティングガイドを作成（`sdd/docs/guides/troubleshooting.md`を参考に）

**目標**: 仕様を更新することがチームの日常になる

## 第5〜8週：プロセス改善

習慣化ができたら、次はプロセスを改善していきます。規模に応じた承認フローを確立し、自動チェックを導入することで、仕様駆動開発の品質を向上させましょう。

- [ ] 承認フローを確立（規模に応じて1〜2名承認）
  - 小規模チーム（3〜5人）：1名承認
  - 中規模プロジェクト（6〜30人）：2名承認
  - 詳細は第6章「規模別に合意形成と承認を実現する方法」を参照
- [ ] Pull Requestテンプレートを作成
- [ ] CI/CDで自動チェック（リンク切れ、文法）を導入
  - `sdd/docs/tools/scripts.md`を参考に

**目標**: 仕様駆動開発のプロセスが安定して回るようになる

## 第9〜12週：拡大と定着

プロセスが安定してきたら、最後は拡大と定着を目指します。トレーサビリティマトリクスを生成し、成功指標を測定することで、仕様駆動開発の効果を可視化しましょう。

- [ ] トレーサビリティマトリクスを生成
  - `sdd/docs/tools/scripts.md`の「トレーサビリティマトリクス生成スクリプト」を参考に
- [ ] 成功指標（KPI）を測定
  - 仕様の更新頻度
  - Pull Requestの数と品質
  - チームメンバーの満足度
- [ ] 他のチームへの展開を検討
- [ ] 振り返り会を開催し、改善点を洗い出す

**目標**: 仕様駆動開発がチームの「当たり前」になり、他のチームへの展開も視野に入れる

## ゴール

90日後には、仕様駆動開発がチームの「当たり前」になっている状態を目指します。完璧を求めず、段階的に実践を積み重ねていくことが重要です。

## 注意事項

- このプランは目安です。チームの状況に応じて、柔軟に調整してください

- うまくいかなくても、諦めずに再開することが大切です
