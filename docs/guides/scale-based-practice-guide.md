# 規模別実践ガイド

このドキュメントは、『仕様駆動開発 実践入門』で使用する規模別の実践ガイドです。本書全体を通じて一貫して使用される規模の定義と、各規模での実践ポイントをまとめています。

## 規模の定義

本書では、以下の3つの規模を定義します。**この定義は、本書全体を通じて一貫して使用されます**。

| 規模 | 人数 | 合意形成の特徴 | 承認の特徴 |
|:---|:---|:---|:---|
| **小規模チーム** | 3〜5人 | 口頭 + Issue（形式は最小限） | 1名承認でOK |
| **中規模プロジェクト** | 6〜30人 | Issue必須（テンプレート活用） | 複数名承認 + 定例レビュー |
| **大規模組織** | 50人以上 | Issue + 稟議・契約との連携 | 階層的承認 + 監査対応 |

**迷ったら「中規模」から始める**：自分のチームがどこに該当するか迷った場合は、まず「中規模プロジェクト」の運用から始めてください。運用が重いと感じたら小規模へ、統制が必要と感じたら大規模へ寄せていくのが現実的です。

詳細は、本書の第6章6.4.1節「規模の定義」を参照してください。

## 規模別の実践ポイント一覧

以下の表は、各観点における規模別の実践ポイントをまとめたものです。

| 観点 | 小規模（3-5人） | 中規模（6-30人） | 大規模（50人以上） |
|------|-----------------|------------------|-------------------|
| **承認プロセス** | 1名承認、セルフマージ可 | 2名承認、CODEOWNERS活用 | 階層的承認、監査対応 |
| **仕様の粒度** | README.md中心、シンプル | 機能ごとにファイル分割 | 詳細なディレクトリ構造 |
| **レビュー** | 日次での確認（5分程度） | 全員参加の週次レビュー会 | 専門チームによるレビュー |
| **ツール** | GitHub標準機能で十分 | CI/CD、自動チェック導入 | GitGuardian、監査ツール必須 |
| **導入期間** | 1週間 | 1ヶ月 | 3ヶ月〜6ヶ月 |

## 各章での規模別のポイント

本書の各章では、規模別の実践ポイントが説明されています。以下に、各章での規模別のポイントへのリンクをまとめます。

### 第3章「Cursorで仕様を記述・活用する」

- **小規模チーム（3〜5人）**：
  - 基本的な.cursorrulesで十分（3.9.3節参照）
  - 全員が同じ設定を共有
  - シンプルなプロンプトで開始

- **中規模プロジェクト（6〜30人）**：
  - より詳細な.cursorrulesを検討（3.9.5節参照）
  - チームごとに設定をカスタマイズ
  - プロンプトテンプレートの共有

- **大規模組織（50人以上）**：
  - プロンプト集の詳細な設定例を参照（`docs/tools/prompts.md`）
  - 組織全体で統一された設定を検討
  - レビューガイドラインの整備

詳細は、本書の第3章3.11節「規模別のポイント」を参照してください。

### 第5章「最初の1週間で仕様駆動開発を始める方法」

- **小規模チーム（3〜5人）**：
  - 全員で同じREADME.mdを共有し、対話しながら更新
  - 1週間で基本的な流れを体感できる
  - 承認プロセスは簡素化

- **中規模プロジェクト（6〜30人）**：
  - チーム代表者を決めて開始
  - 週次レビュー会を設ける
  - 承認プロセスは2名承認を検討

- **大規模組織（50人以上）**：
  - 特定のチームやプロジェクトから開始（第11章11.7節参照）
  - 段階的な導入を検討
  - 経営層の理解を得る（第11章11.6節参照）

詳細は、本書の第5章5.8節「規模別のポイント」を参照してください。

## 規模別の移行パス

仕様駆動開発は、小規模から始めて段階的に拡大することを推奨します。以下は、規模を拡大する際の移行パスです。

### 小規模（3-5人）→ 中規模（6-30人）への移行

**移行のタイミング**：
- チームメンバーが6人以上になった
- 複数の機能を同時に開発するようになった
- 1名承認では判断が難しくなってきた

**移行時の注意点**：
- [ ] CODEOWNERSファイルの導入を検討
- [ ] 承認プロセスを1名から2名に変更
- [ ] 仕様ファイルを機能ごとに分割する準備
- [ ] CI/CDの導入を検討
- [ ] 定例レビュー会の導入を検討

**移行のステップ**：
1. まず、CODEOWNERSファイルを導入し、承認プロセスを2名に変更
2. 仕様ファイルを機能ごとに分割する準備を始める
3. CI/CDの導入を検討し、自動チェックを導入
4. 定例レビュー会を設け、チーム全体での合意形成を強化

### 中規模（6-30人）→ 大規模（50人以上）への移行

**移行のタイミング**：
- 組織全体で50人以上が関わるようになった
- 複数の部署やチームが連携するようになった
- 監査対応や契約対応が必要になった

**移行時の注意点**：
- [ ] 階層的な承認フローの導入
- [ ] セキュリティツール（GitGuardianなど）の導入
- [ ] 監査証跡の整備
- [ ] 専門チームによるレビューの導入
- [ ] 稟議・契約との連携プロセスの整備

**移行のステップ**：
1. まず、階層的な承認フローを導入し、承認プロセスを明確化
2. セキュリティツールを導入し、セキュリティリスクを軽減
3. 監査証跡を整備し、監査対応を強化
4. 専門チームによるレビューを導入し、品質を向上
5. 稟議・契約との連携プロセスを整備し、組織的な運用を確立

詳細は、本書の第6章6.8節「規模別の移行パス」を参照してください。また、組織への展開方法については、第11章「仕様駆動開発の実践的進め方」、第12章「組織全体導入のガバナンス設計」、第13章「組織全体導入の文化変革」を参照してください。
