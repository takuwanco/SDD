## 1. 個人情報を仕様に含めない設計パターン

仕様書に個人情報を含めると、以下のリスクが発生します：

- **法的リスク**：個人情報保護法、GDPRなどの法規制違反
- **セキュリティリスク**：リポジトリが漏洩した場合の情報漏洩
- **監査リスク**：監査時に問題とされる可能性

### 1.1 NG例：個人情報を含む仕様

```markdown
## テストデータ
- ユーザー名: 山田太郎
- メールアドレス: yamada@example.com
- 電話番号: 090-1234-5678
- 住所: 東京都千代田区...
```

**問題点**：
- 実在する個人情報が含まれている
- リポジトリが漏洩した場合、個人情報保護法違反の可能性
- 監査時に問題とされる可能性

### 1.2 OK例：個人情報を含まない仕様

```markdown
## テストデータ
- ユーザー名: <テスト用のダミーデータを使用>
- メールアドレス: test-{random}@example.com（ランダム生成）
- 電話番号: 000-0000-0000（ダミー）
- 住所: テスト用のダミー住所を使用

**注意**：
- 実際のユーザー情報は、環境変数またはシークレット管理ツールで管理する
- 仕様書には含めない
- テストデータは、テスト実行時に動的に生成する
```

**推奨事項**：
- テストデータは、テスト実行時に動的に生成する
- 実際のユーザー情報が必要な場合は、環境変数またはシークレット管理ツール（AWS Secrets Manager、Azure Key Vaultなど）で管理する
- 仕様書には、データの形式や構造のみを記載し、実際の値は含めない

### 1.3 個人情報を含む可能性がある情報の例

以下の情報は、個人情報に該当する可能性があるため、仕様書に含めないように注意してください：

- **氏名**：実在する人物の氏名
- **メールアドレス**：実在するメールアドレス
- **電話番号**：実在する電話番号
- **住所**：実在する住所
- **生年月日**：実在する人物の生年月日
- **個人識別番号**：マイナンバー、パスポート番号など
- **医療情報**：診断名、処方薬など
- **金融情報**：口座番号、クレジットカード番号など

## 2. プライベートリポジトリとパブリックリポジトリの使い分け

仕様駆動開発では、仕様とコードを同一リポジトリで管理しますが、情報の種類に応じてリポジトリの公開設定を適切に選択する必要があります。

### 2.1 リポジトリタイプの選択基準

| 情報の種類 | リポジトリタイプ | 理由 | 注意点 |
|------------|------------------|------|--------|
| **一般的な仕様（APIエンドポイント、画面遷移など）** | パブリック可 | 公開しても問題ない | ただし、ビジネスロジックの詳細は含めない |
| **ビジネスロジックの詳細** | プライベート | 競合に知られたくない | 組織の機密情報ポリシーに従う |
| **セキュリティ要件（認証方式、暗号化方式）** | プライベート | 攻撃の手がかりになる | セキュリティ設計の詳細は特に注意 |
| **個人情報を含むテストデータ** | 絶対にプライベート | 法的リスク | 個人情報保護法、GDPRなどの法規制違反の可能性 |
| **監査証跡** | プライベート | 組織の内部情報 | 監査記録は機密情報として扱う |
| **稟議・承認プロセス** | プライベート | 組織の内部情報 | 承認フローの詳細は機密情報 |

### 2.2 ハイブリッドアプローチ

大規模組織では、以下のようなハイブリッドアプローチが有効です：

- **パブリックリポジトリ**：一般的な仕様、API仕様、画面遷移図など
- **プライベートリポジトリ**：ビジネスロジックの詳細、セキュリティ要件、個人情報を含むテストデータなど

パブリックリポジトリからプライベートリポジトリへの参照を記載することで、公開可能な情報と機密情報を適切に分離できます。

## 3. シークレット検出ツールの比較

シークレット（APIキー、パスワード、トークンなど）が誤ってコミットされないように、シークレット検出ツールを導入することが重要です。

### 3.1 シークレット検出ツールの比較表

| ツール | 特徴 | 無償版 | 有償版 | 推奨用途 | 導入難易度 |
|--------|------|--------|--------|----------|------------|
| **GitGuardian** | 高精度、GitHub連携強力、ダッシュボード機能 | ◯（個人・小規模） | ◯ | 中〜大規模組織 | 低（GitHubマーケットプレースから簡単に導入可能） |
| **git-secrets** | AWS公式、軽量、コマンドライン | ◯（完全無償） | - | AWSを使う小規模チーム | 中（コマンドラインでの設定が必要） |
| **TruffleHog** | オープンソース、カスタマイズ可、CI/CD統合 | ◯ | - | 技術者向け | 中（CI/CDパイプラインへの統合が必要） |
| **GitHub Secret Scanning** | GitHub組み込み、自動検出 | ◯（パブリックリポジトリ） | ◯（プライベート） | GitHub利用者全般 | 低（GitHubの設定のみ） |

### 3.2 ツールの選定基準

**小規模チーム（3〜5人）**：
- **推奨**：GitHub Secret Scanning（プライベートリポジトリの場合、有償版が必要）
- **代替**：git-secrets（AWSを使用している場合）

**中規模プロジェクト（6〜30人）**：
- **推奨**：GitGuardian（無償版で基本的な検出機能を利用可能）
- **代替**：GitHub Secret Scanning + TruffleHog

**大規模組織（50人以上）**：
- **推奨**：GitGuardian（有償版で高度な機能を利用）
- **補完**：TruffleHog（CI/CDパイプラインへの統合）

### 3.3 GitGuardianの導入例

GitGuardianは、GitHubマーケットプレースから簡単に導入できます。

1. GitHubリポジトリの「Settings」→「Integrations」→「GitHub Apps」を開く
2. 「GitGuardian」を検索してインストール
3. リポジトリを選択して有効化

GitGuardianは、コミットされたシークレットを自動的に検出し、アラートを送信します。

詳細は、本書の第12章12.5.4節「方法3：シークレット検出ツールの導入」を参照してください。

## 4. 監査ログの長期保存

GitHubの履歴は永続的に保存されますが、組織のポリシーで「10年以上の保存」が必要な場合があります。また、法的要件で監査証跡の長期保存が求められる場合もあります。

### 4.1 定期的なアーカイブ

リポジトリ全体を定期的にアーカイブすることで、長期保存を実現できます。

```bash
# 毎年1月1日にリポジトリ全体をアーカイブ
git bundle create archive-2025.bundle --all

# アーカイブの確認
git bundle verify archive-2025.bundle
```

**推奨頻度**：
- 年次：毎年1月1日に前年のアーカイブを作成
- 四半期：四半期ごとにアーカイブを作成（より頻繁なバックアップが必要な場合）

### 4.2 外部バックアップ

アーカイブを外部ストレージに保存することで、災害対策にもなります。

**推奨ストレージ**：
- **AWS S3**：S3 Glacierに保存することで、長期保存のコストを削減
- **Azure Blob Storage**：Azure Archive Storageに保存することで、長期保存のコストを削減
- **Google Cloud Storage**：Coldline Storageに保存することで、長期保存のコストを削減

**暗号化**：
- アーカイブを暗号化して保存（法的要件に対応）
- 暗号化キーは、シークレット管理ツールで管理

### 4.3 監査証跡の出力

監査時に必要な情報をCSV形式で出力します。

```python
#!/usr/bin/env python3
# scripts/generate_audit_log.py

import subprocess
import csv
from datetime import datetime

def generate_audit_log(repo_path, output_file):
    """監査用のレポートを生成"""
    
    # Gitログを取得
    cmd = [
        'git', '-C', repo_path, 'log',
        '--all',
        '--format=%H,%an,%ae,%ad,%s',
        '--date=iso'
    ]
    
    result = subprocess.run(cmd, capture_output=True, text=True, check=True)
    
    # CSVファイルに書き込み
    with open(output_file, 'w', encoding='utf-8') as f:
        writer = csv.writer(f)
        writer.writerow(['コミットハッシュ', '作成者名', 'メールアドレス', '日時', 'コミットメッセージ'])
        
        for line in result.stdout.strip().split('\n'):
            if line:
                writer.writerow(line.split(',', 4))
    
    print(f"✅ 監査ログを生成しました: {output_file}")

if __name__ == '__main__':
    import sys
    repo_path = sys.argv[1] if len(sys.argv) > 1 else '.'
    output_file = sys.argv[2] if len(sys.argv) > 2 else 'audit_log.csv'
    generate_audit_log(repo_path, output_file)
```

**使用方法**：
```bash
python scripts/generate_audit_log.py . audit_log_2025.csv
```

### 4.4 監査証跡に含めるべき情報

監査証跡には、以下の情報を含めることを推奨します：

- **コミットハッシュ**：変更の一意な識別子
- **作成者名**：変更を行った人の名前
- **メールアドレス**：変更を行った人のメールアドレス
- **日時**：変更が行われた日時（ISO 8601形式）
- **コミットメッセージ**：変更の理由や内容
- **変更されたファイル**：どのファイルが変更されたか
- **Pull Request番号**：関連するPull Requestの番号
- **承認者**：Pull Requestを承認した人の名前

### 4.5 監査証跡の保存期間

組織のポリシーや法的要件に応じて、監査証跡の保存期間を決定します。

**一般的な保存期間**：
- **金融機関**：7年〜10年
- **医療機関**：5年〜7年
- **公的機関**：10年以上
- **一般企業**：3年〜5年

保存期間を超えた監査証跡は、組織のポリシーに従って削除またはアーカイブします。

## 5. GDPR、個人情報保護法への対応

GDPR（EU一般データ保護規則）や個人情報保護法に対応するため、以下の点に注意してください。

### 5.1 個人情報の取り扱い

- **最小化の原則**：必要最小限の個人情報のみを収集・利用する
- **目的外利用の禁止**：収集した個人情報は、収集時の目的以外に利用しない
- **適切な管理**：個人情報は適切に管理し、漏洩を防止する

### 5.2 仕様書での個人情報の取り扱い

- **個人情報を含めない**：仕様書には個人情報を含めない（1.1節、1.2節を参照）
- **匿名化・仮名化**：テストデータなどで個人情報が必要な場合は、匿名化または仮名化する
- **アクセス制御**：個人情報を含む可能性がある情報は、プライベートリポジトリで管理する

### 5.3 データ主体の権利への対応

GDPRや個人情報保護法では、データ主体（個人情報の本人）に以下の権利が認められています：

- **アクセス権**：自分の個人情報へのアクセスを要求する権利
- **訂正権**：不正確な個人情報の訂正を要求する権利
- **削除権**：個人情報の削除を要求する権利
- **ポータビリティ権**：個人情報の移転を要求する権利

仕様駆動開発では、Gitの履歴により変更履歴が完全に記録されるため、データ主体の権利への対応が容易になります。

## 6. 規制産業向けの追加考慮事項

### 6.1 金融機関向け

- **金融機関検査マニュアル**：金融機関検査マニュアルに準拠した監査証跡の保存
- **内部統制**：内部統制の観点から、変更履歴の完全な記録が必要
- **リスク管理**：リスク管理の観点から、セキュリティ要件の明確化が必要

### 6.2 医療機関向け

- **医療情報の保護**：診断名、処方薬などの医療情報は、特に注意して取り扱う
- **患者情報の匿名化**：テストデータなどで患者情報が必要な場合は、必ず匿名化する
- **アクセスログの保存**：医療情報へのアクセスログを適切に保存する

### 6.3 公的機関向け

- **情報公開法への対応**：情報公開法に基づく開示請求に対応できるよう、情報を適切に管理する
- **長期保存**：公的機関では、10年以上の長期保存が必要な場合がある
- **監査対応**：監査院などの監査に対応できるよう、監査証跡を適切に保存する

## 8. まとめ

規制産業や公的機関で仕様駆動開発を導入する際は、以下の点に注意してください：

1. **個人情報を仕様に含めない**：個人情報保護法、GDPRなどの法規制違反を防ぐ
2. **リポジトリの公開設定を適切に選択**：情報の種類に応じて、プライベート/パブリックを適切に選択
3. **シークレット検出ツールを導入**：APIキー、パスワードなどのシークレットが誤ってコミットされないようにする
4. **監査証跡を適切に保存**：組織のポリシーや法的要件に応じて、監査証跡を長期保存する

これらの対策により、規制産業や公的機関でも、仕様駆動開発を安全に導入できます。
