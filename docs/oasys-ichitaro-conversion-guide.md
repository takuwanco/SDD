# OASYS/一太郎からMarkdownへの変換ガイド

本ガイドは、書籍の補足として「手を動かす人」向けに具体化した **実践リファレンス** です。  
ここでのゴールは、**OASYS/一太郎の資産を"仕様として育てられる形"に再生**することです。

> **重要**：変換の目的は「見た目の再現」ではなく、**構造（見出し・段落・表・関係性）の回復**です。  
> まず"使える状態"を作り、品質は運用の中で磨きます。

## 1. まず最初に：このガイドの使い方

### 3つの読み方
1. **1回だけ変換したい**：3 → 4 → 5/6/7 → 11 だけ読む  
2. **継続運用したい**：2 → 4 → 8 → 9 を重点的に読む  
3. **詰まった**：11 を辞書的に引く  

### 先に決めること（迷いを減らす）
- 変換対象は「今も更新される文書」か？（更新されないならPDF保管＋索引でもよい）
- 目的は「チーム参照」か？「外部公開」か？（必要品質が変わる）
- 機密情報を含むか？（オンラインツールの可否が変わる）
- 元のファイルが開けるか？（OASYS Viewer、一太郎ビューアの有無）

## 2. 変換の基本原則（失敗しない判断軸）

### 原則1：変換は"再現"ではなく"回復"
- 見出し（階層）
- 箇条書き（意図）
- 表（列の意味）
- 図面（要素間の関係性）
- 参照（リンク・画像）

この **構造が残っていれば合格** です。装飾は後から直せます。

### 原則2：最初から完璧を目指さない
最低限、次の3点を満たせば「仕様として再生可能」です。
- 見出し構造が把握できる
- 表や項目が意味を失っていない
- 重要情報が欠落していない

### 原則3：段階的に進める
すべてを一度にやろうとせず、優先度の高い文書から着手します。
1. **第一歩：PDF化・画像化**（文書を失わない）
2. **第二歩：構造抽出**（図面・段落・スタイル）
3. **第三歩：Markdown化**（Git管理・AI連携）

### 原則4：ツールより運用が大事
「誰が」「いつ」「どうやって更新し」「差分をどう確認するか」。  
これが決まっていないと、変換だけして終わります。

## 3. ビューワーでPDF化・画像化（実務的対処）

理想を言えば、すべての文書をMarkdown形式に変換したいところですが、現実的には数百、数千という単位で残っている文書すべてを、いきなり完璧に移行することは困難です。

そこで、まず取り組むべきは**「文書を失わないこと」**です。そして、その第一歩として最も現実的な方法が、**閲覧専用ビューワーを用いてPDF化・画像化すること**です。

### 3.1 なぜPDF化・画像化が現実的なのか

PDF化や画像化は、一見すると「後ろ向きな対処」に思えるかもしれません。しかし、次のような理由から、現場での第一歩として非常に有効です。

1. **技術的なハードルが低い**：ビューワーで開いて印刷（PDF出力）するだけで完了する
2. **原本の見た目を保持できる**：レイアウトや書式が崩れることなく保存できる
3. **閲覧環境を選ばない**：PDFはほぼすべてのデバイスで開ける
4. **時間的な猶予を得られる**：完全な移行計画を立てる前に、とりあえず「失わない」状態を作れる

つまり、PDF化・画像化は「完璧な解決策」ではなく、「現実的な第一歩」なのです。

### 3.2 OASYS文書の場合

OASYS文書の場合、ビューワーの公開が終了しているため、選択肢は極めて限られています。

**方法1：古い環境を維持してPDF化**
- もし組織内に、過去に入手した「OASYS Viewer」が残っており、かつWindows XP環境（または仮想環境）を用意できる場合は、PDF化が可能です
- ただし、Windows XP環境の維持にはセキュリティリスクが伴います

**方法2：印刷物をスキャンして画像化**
- 元のファイルが開けない場合、あるいはファイル自体が失われている場合は、印刷物をスキャンして画像化（PDF化）する方法があります
- OCR（光学文字認識）機能付きスキャナーを使えば、検索可能なPDFを作成できます

**方法3：専門業者への依頼**
- 組織内にOASYS Viewerや古い環境がない場合、文書変換を専門とする業者に依頼する方法もあります

### 3.3 一太郎文書の場合

一太郎文書の場合は、現在も無償の「一太郎ビューア」が提供されているため、OASYS文書と比べて対応は容易です。

**手順：**
1. 一太郎ビューアをインストール（ジャストシステムの公式サイトから無償ダウンロード）
2. 対象の一太郎ファイルを開く
3. 「ファイル」→「印刷」→「Microsoft Print to PDF」でPDF出力

Windows 10以降であれば、「Microsoft Print to PDF」が標準で搭載されているため、追加のソフトウェアは不要です。

### 3.4 この方法の限界

ただし、PDF化・画像化には明確な限界もあります。

- **検索性の低下**：画像化した文書では、全文検索ができません。OCR処理を行ったPDFでも、検索精度には限界があります
- **再編集性の喪失**：PDFや画像では、内容の修正や更新ができません。誤字を見つけても直せず、仕様変更があっても反映できません
- **構造情報の欠落**：見出し、段落、箇条書きといった文書の構造情報が失われます。これは、後でAIを使って要約や分析を行う際に大きな障害となります
- **バージョン管理の困難**：PDFファイルはバイナリ形式なので、Gitなどのバージョン管理システムで差分管理ができません

これらの限界を踏まえると、PDF化・画像化は**「保存」のための手段であり、「活用」のためには次のステップが必要**だとわかります。

## 4. 構造抽出の基本方針

OASYSや一太郎の文書から構造を取り出すには、二つのアプローチがあります。

**アプローチ1：元の形式のまま構造情報を読み取る**
- 一太郎の場合：製品版の一太郎やJUST PDF、またはLibreOffice等で開き、構造情報を活用
- OASYSの場合：OASYS Viewerや古い環境で開き、可能な限り構造情報を保持して変換

**アプローチ2：Word形式を経由して構造を復元する**
- 一太郎 → Word → Markdownのルートで、Word形式が持つスタイル情報を活用
- この場合、Word形式での「構造を意識した文書作成」の原則が重要になる

Word形式を間に挟むことのメリットが不明だと思う読者もいらっしゃるかもしれませんが、顧客やビジネスパートナー、社内説明など、Word文書が存在することで、現状の日本のビジネスシーンでは役立つ現実的で、役立つアプローチです。

### 4.1 構造を失うことの深刻さ

OASYSや一太郎などの文書には、図面や段落、スタイルといった **構造的な情報が内部的に保持されています**。これらを単純にPDF化や画像化してしまうと、見た目は保てても **「構造」そのものが失われてしまう** ことは前述しました。

たとえば、元の文書では
- 「第1章」が見出しレベル1として定義されている
- 箇条書きが階層構造を持っている
- 図面内の要素がグループ化され、関係性を持っている

といった情報が、PDF化や画像化によって
- 「第1章」が単に **「大きめのフォントで書かれたテキスト」** になる
- 箇条書きが「・」という**記号を含む普通のテキスト行**になる
- 図面が一枚の画像（ビットマップ）になり、**要素間の関係性が消える**

このような状態では、AIによる要約や分析、Markdownへの自動変換は極めて困難になります。なぜなら、AIは **「構造」を手がかりに文書を理解する** からです。

構造を抽出することは、 **AI変換や再利用のための第一歩** となるのです。

## 5. 図面の抽出 ― 関係性を保持する

文書に含まれる図面やイラストは、単なる画像として扱ってしまうと、その価値を大きく損なってしまいます。なぜなら、図面は **「関係性を持つ情報」** だからです。

### 5.1 図面が持つ「関係性」とは

たとえば、システム構成図を例にとると
- サーバーとクライアントが矢印で結ばれ、通信関係を示している
- データベースとアプリケーションサーバーが隣接し、データの流れを表現している
- ファイアウォールが境界に配置され、セキュリティ境界を示している

これらの要素を単なる画像として扱うと、「四角形が三つと矢印が二つある絵」になってしまい、本来の意味が失われます。

### 5.2 図面抽出の実践手法

**手法1：グループ化された図形をまとめて抽出する**

OASYSや一太郎で作成された図形は、多くの場合、意味のある単位でグループ化されています。例えば
- 「サーバー」という四角形と、その中の「Webサーバー」というテキスト
- 複数のステップを示す矢印付きフローチャート
- 凡例（色や記号の説明）

これらをバラバラにするのではなく、 **グループ単位で抽出** することで、意味を保持できます。

**実践的なアプローチ：**
1. 一太郎の場合：一太郎で文書を開き、図形を選択して「図形の保存」で個別に保存
2. Word経由の場合：Wordで開き、図形を右クリック→「図として保存」でPNG形式で保存
3. OASYS形式の場合：可能であればOASYS Viewerで開き、スクリーンショットで切り取る

**手法2：図中のテキストを構造として取り出す**

図面には、ラベルや説明テキストが含まれています。これらは、OCRで読み取るのではなく、 **構造解析ツールで取り出す** ことが理想です。

**具体的なツール：**
- **LibreOffice Draw**：一太郎やWord文書をインポートし、図形を個別に編集・抽出できる
- **Inkscape**：ベクター形式（SVG）で図形を扱え、テキスト情報を保持したまま編集可能
- **変換ツール**：一太郎の製品版には、図形をSVG形式で出力する機能がある

これらのツールを使うと、図中のテキストを「画像の一部」ではなく「編集可能なテキスト」として取り出せます。

**手法3：関係線や矢印を削除せず保持する**

システム構成図やフローチャートでは、要素間を結ぶ矢印や線が重要な意味を持ちます。これらを含めて図面を保存することで、 **要素の依存関係を保持** できます。

### 5.3 図面の再利用を考えた保存方法

抽出した図面は、次のような構造で整理すると良いでしょう。

```
project/
├── docs/
│   └── specification.md
└── images/
    ├── diagrams/           # 図面（構成図、フロー図など）
    │   ├── system-architecture.png
    │   └── data-flow.svg
    ├── screenshots/        # スクリーンショット
    │   └── ui-mockup.png
    └── legacy/             # 元の文書から抽出した画像
        └── oasys-fig-01.png
```

**実践的なアドバイス：**
- 図面をPNG形式で保存する際は、高解像度（300dpi以上）を維持
- 可能であればSVG形式で保存し、将来の編集可能性を残す
- 図面ごとに説明文をMarkdown内に記述し、意味を補完する

高解像度を維持するのは、マルチモーダルなAIを使って図形を読ませることになった場合に、有効であるためです。またSVG形式は、「Scalable Vector Graphics（スケーラブル・ベクター・グラフィックス）」の略で、拡大・縮小しても画質が劣化しない、Webに適したベクター形式の画像ファイルです。XMLベースのテキストデータであることから、AIにも比較的扱いやすい形式です。

Markdown内では以下のように記載しておきます。

```markdown
---

![システム構成図](images/diagrams/system-architecture.png)

*図：システム構成図（Webサーバー、アプリケーションサーバー、データベースの3層構成）*

---
```

## 6. 段落構造の抽出 ― 論理的な階層を復元する

OASYSや一太郎、そして多くのWord文書では、段落区切りが **単なる改行で表現されている** ことがあります。これを正しく抽出し、論理的な構造として復元することが重要です。ここでも手作業が発生する可能性があります。

### 6.1 段落構造とは何か

以下のような「段落構造」とは、文書が持つ論理的な階層のことです。

```
第1章 概要（レベル1の見出し）
  1.1 背景（レベル2の見出し）
    本システムは...（本文段落）
    以下の課題があった：（本文段落）
      ・課題1（箇条書き）
      ・課題2（箇条書き）
  1.2 目的（レベル2の見出し）
    上記課題を解決するため...（本文段落）
```

このような形式・状態を「ネストする」と表現することがあります。ネストするとは、「入れ子（いれこ）にする」という意味で、ある構造の中に同じような構造を内包することを指します。プログラムのコードでよく使われるものですが、Markdownを書くときには、このネストする構造を意識するといいでしょう。

この構造をMarkdown形式で表現すると、以下の通りです。

```markdown
# 1 第1章 概要

本システムは...

以下の課題があった：

- 課題1
- 課題2

## 1.2 目的

上記課題を解決するため...
```

### 6.2 段落構造抽出の3つのステップ

**ステップ1：見出し・本文・箇条書きを識別する**

多くの文書では、スタイル情報によって要素の種類が定義されています。

- **OASYSの場合**：段落スタイルとして「標題」「見出し」「本文」などが定義されている
- **一太郎の場合**：「段落スタイル」として「見出し1」「見出し2」「本文」などが設定されている
- **Word経由の場合**：Wordの「見出し1」「見出し2」「標準」などのスタイルが適用されている

これらのスタイル情報を活用して、Markdownの記法にマッピングします。

| 元の文書のスタイル | Markdown記法 |
|-------------------|--------------|
| 見出し1、大見出し | `#` |
| 見出し2、中見出し | `##` |
| 見出し3、小見出し | `###` |
| 本文、標準 | （通常の段落） |
| 箇条書き | `- ` または `1. ` |
| 引用 | `> ` |

**ステップ2：改行の種類を区別する**

ワープロソフトでは、二種類の改行があります。

- **段落区切り（Enter）**：論理的な段落の終わり
- **行内改行（Shift+Enter）**：同じ段落内での見た目上の改行

Markdown形式では、この区別が重要です。

```markdown
これは一つの段落です。
この行は前の行と同じ段落内です。

これは新しい段落です。
```

OASYSや一太郎からの変換時には、この区別が失われやすいため、注意が必要です。

**実践的なアプローチ：**
- Word経由で変換する場合、Wordで一度開いて段落区切りを確認
- 不要な空行（連続する改行）は削除
- 意味のある段落区切りは保持

**ステップ3：論理的な階層を確認する**

変換後、文書の階層構造が論理的に保たれているか確認します。

**チェックポイント：**
- 見出しのレベルが飛んでいないか（`#`の次にいきなり`###`になっていないか）
- 箇条書きが適切にインデントされているか
- 本文と見出しが混在していないか

**確認方法：**
多くのMarkdownエディタ（VS Code、Cursor、Typora、Obsidianなど）には、アウトライン表示機能があります。これを使って、文書の階層構造を視覚的に確認できます。

### 6.3 段落構造抽出における実践的な注意点

**注意点1：「番号付き見出し」の扱い**

日本の文書では、見出しに番号が含まれることが多くあります。

```
1. 概要
1.1 背景
1.2 目的
2. 要件
2.1 機能要件
```

また文章中に、見出しの番号が含まれる場合もあるでしょう。
これをMarkdown形式にする際、番号の扱いには選択肢があります。

**選択肢A：番号を見出しに含める**
```markdown
# 1. 概要
# 2. 要件
```

**選択肢B：番号を除いて見出しレベルのみ表現**
```markdown
# 概要
# 要件
```

組織の規約や、文書の性質に応じて選択してください。前述した通り、文章中に、見出し番号が記載されている（「べた書き」と呼ばれます）場合、本文の修正なども必要になります。また一貫性を保つことも重要です。

**注意点2：表の段落構造**

表内のセルにも段落構造が存在します。特に、セル内に箇条書きがある場合、これを正しく抽出する必要があります。
Markdown形式の表内では、複雑な構造を表現しにくいため、次のような工夫が必要です。

```markdown
| 機能 | 説明 |
|------|------|
| 機能A | ・要件1<br>・要件2<br>・要件3 |
```

または、表を分割して、箇条書きは表の外に出す。

```markdown
| 機能 | 説明 |
|------|------|
| 機能A | 以下の要件を満たす |

**機能Aの要件：**
- 要件1
- 要件2
- 要件3
```

複雑なテーブルは、分割するなどして、シンプルなテーブルにするほうが後々メンテナンスも楽になります。

## 7. スタイル情報の抽出 ― 構造の手がかりを活かす

文書のスタイル設定には、 **構造の手がかりが多く含まれます**。これらを正しく読み取り、Markdown形式に反映させることが重要です。筆者は、この作業が人手で行う際にもっとも手間がかかるのではないかと考えています。

### 7.1 スタイル情報とは何か

「スタイル情報」とは、文書の見た目を定義する設定のことです。

- **文字スタイル**：フォント種類、サイズ、太字、斜体、色など
- **段落スタイル**：行間、インデント、配置（左揃え、中央揃えなど）
- **見出しスタイル**：階層レベル、番号付けルールなど

これらの中で、Markdown形式への変換において重要なのは、 **意味を持つスタイル** です。

### 7.2 スタイル抽出の3つの原則

**原則1：階層スタイルを見出し構造として再現**

OASYSや一太郎では、「見出し1」「見出し2」といった階層的なスタイルが定義されています。これらは、Markdownの見出し記法に直接対応します。

| 元の文書 | Markdown |
|---------|----------|
| 見出し1 | `# 見出し` |
| 見出し2 | `## 見出し` |
| 見出し3 | `### 見出し` |
| 見出し4 | `#### 見出し` |

**実践的なアプローチ：**
1. 文書を開き、各見出しに適用されているスタイルを確認
2. スタイル名と見出しレベルの対応表を作成
3. 変換ツールやスクリプトで一括変換

**原則2：装飾を意味に応じてMarkdown記法に置き換える**

太字、斜体、下線などの装飾は、見た目だけでなく **意味** を持っています。この意味を理解し、適切にMarkdown記法に変換します。

| 装飾の種類 | 意味 | Markdown記法 |
|-----------|------|-------------|
| 太字 | 強調、重要なキーワード | `**太字**` |
| 斜体 | 用語の定義、引用 | `*斜体*` |
| 等幅フォント | コード、コマンド | `` `code` `` |
| 下線 | （Markdownでは非推奨） | 太字に置き換える |

**実践例：**

**元の文書：**
```
本システムの目標は、処理時間を50%削減することである。
```
（「50%削減」が太字で強調）

**Markdown形式：**
```markdown
本システムの目標は、処理時間を**50%削減**することである。
```

**原則3：意味を持たない装飾は省略して構造に集中**

すべてのスタイル情報を保持する必要はありません。特に、次のような装飾は **省略してかまいません**。例えば以下のような情報ですが、一度、削除をしてしまうと、元に戻すことが難しいので、組織的にルールを決めてから取り組みましょう。

- **フォントの種類**：Markdown表示環境で自動的に適切なフォントが選ばれる
- **フォントサイズ**：見出しレベルでサイズは表現される
- **文字色**：特別な意味を持たない色付けは不要
- **行間・余白**：Markdown表示環境で適切に設定される

**例：色情報の扱い**

1. 元の文書で「重要」という言葉が赤色で書かれていた場合

```
重要：必ず管理者権限で実行してください。
```
（「重要」が赤色）

2. Markdown形式では、色ではなく太字や記号で表現

```markdown
**重要**：必ず管理者権限で実行してください。
```

```markdown
> **重要**
> 必ず管理者権限で実行してください。
```

このように、「赤色」という視覚的な装飾を、「太字」や「引用ブロック」という構造的な表現に置き換えることで、意味を保持しつつシンプルな形式にできます。

### 7.3 スタイル情報の抽出における実践的なワークフロー

**ステップ1：スタイル一覧の作成**

まず、元の文書で使用されているスタイルを一覧化します。

```
- 大見出し → Markdownの`#`
- 中見出し → Markdownの`##`
- 小見出し → Markdownの`###`
- 本文 → 通常の段落
- 強調（太字） → `**...**`
- コマンド（等幅） → `\`...\``
```

**ステップ2：変換ルールの適用**

この一覧をもとに、変換ツールやAIに指示を出します。

```
プロンプト例：
以下のWord文書を、次のルールに従ってMarkdown形式に変換してください：
- 「見出し1」スタイル → `#`
- 「見出し2」スタイル → `##`
- 太字の箇所 → `**...**`
- 等幅フォントの箇所 → `\`...\``
- 赤色のテキスト → 太字に変換
```

**ステップ3：手作業での確認と調整**

自動変換後、以下の点を手作業で確認します。

- 見出しレベルが正しく変換されているか
- 強調の意図が保たれているか
- 不要な装飾が除去されているか
- 段落構造が論理的に保たれているか

### 7.4 まとめ ― 構造抽出は「復元作業」

本節では、OASYSや一太郎の文書から、図面、段落構造、スタイル情報を取り出す方法を見てきました。

重要なのは、これが単なる「ファイル変換」ではなく、 **「構造の復元作業」** だという点です。元の文書作成者が意図した論理的な階層、要素間の関係性、強調の意味――これらを理解し、Markdown形式で表現し直すのです。

この作業は、完全に自動化することは困難です。しかし、次節でご紹介するAIの支援を活用することで、大幅に効率化できます。

重要なのは、以下の点です。
- **完璧を目指さない**：80%の精度で変換し、使いながら改善する
- **段階的に進める**：すべてを一度にやろうとせず、優先度の高い文書から着手
- **構造を意識する**：見た目ではなく、意味と構造を保持することに集中

## 8. Word経由の変換（橋渡しアプローチ）

一太郎やOASYSの文書をWord形式に移行することは、多くの組織で現実的な選択肢です。実際、組織の標準がWordになっている現在、Word形式への移行は編集の容易さや社内での共有という観点から、非常に有効な選択です。

しかし、ここで立ち止まって考える必要があります。**Wordへの変換は、本当に「課題の解決」なのでしょうか？**

結論から言えば、Wordへの変換は**「見た目を残す」ための一時的な解決**にすぎません。今後、AIと連携して文書を活用し、仕様駆動開発を実践していくためには、さらに一歩進んだアプローチが必要です。それが、**構造情報を持ったMarkdown形式への移行**です。

### 8.1 Word形式へ移行するなら3つのポイント

1. **バージョン管理を意識した運用設計**：SharePointやTeams、OneDriveを活用することで、Microsoft Office 365のバージョン履歴機能を使うことができます。限定的ではありますが、変更の追跡が可能です。

2. **構造を意識した文書作成**：Wordで作成する文書をAIと連携させたり、他の形式に変換したりする可能性を考えるなら、**「構造を意識した文書作成」が鍵**となります。スタイル機能の活用、箇条書き機能の統一、表機能の適切な使用、段落の明確化が重要です。

3. **将来を見据えた準備**：Word形式で文書を管理する場合でも、「将来、別の形式への移行が必要になる可能性」を念頭に置いた設計が賢明です。エクスポート機能の確認、互換性の維持、メタデータの記録、定期的な見直しが有効です。

### 8.2 Word形式はMarkdownへの橋渡し

筆者は、Word形式は、Markdown形式への最高の橋渡しではないか、とさえ感じることがあります。広く普及したWord形式は、以下の観点からも、非常に優れた選択肢であると言えるでしょう。

- 組織の標準がWordである：既存の文書体系との整合性が保てる
- 編集者全員がWordに習熟している：新しいツールの学習コストがかからない
- 印刷物としての完成度が重要：レイアウトの細かな調整が容易
- Microsoft Office 365を活用している：共同編集やバージョン管理の機能が使える

重要なのは、「形式そのもの」ではなく**「その形式をどう使うか」**なのです。

### 8.3 WordからMarkdownへの変換

Word形式への移行が完了したら、次はMarkdown形式への変換です。WordからMarkdownへの変換方法については、**SDDリポジトリの`docs/word-excel-conversion-guide.md`**を参照してください。

## 9. AI変換支援（段落タグ付け・章構造推定）

前節までで、OASYSや一太郎の文書から図面、段落構造、スタイル情報を取り出す方法を見てきました。しかし、これらの作業を完全に手動で行うのは、特に大量の文書がある場合、現実的ではありません。

ここで強力な支援となるのが、**AIによる構造推定とタグ付け**です。Claude、ChatGPT、Geminiなどの大規模言語モデル（LLM）は、文書の構造を理解し、適切なMarkdown形式への変換を支援する能力を持っています。

### 9.1 AIが得意とする変換支援の3つの領域

#### 9.1.1 領域1：段落の意味分類とタグ付け

AIは文中の文脈を解析し、段落ごとの意味を自動的に分類することができます。たとえば、「目的」「前提」「手順」「結果」「制約条件」といったセクション区分を判別し、適切な見出しレベルやタグを付与します。

**実践例：Claude/ChatGPTへのプロンプト**

```markdown
以下のテキストは、一太郎で作成された古い仕様書から抽出したものです。
このテキストを分析し、各段落の意味（目的、前提条件、手順、制約など）を
判別して、適切なMarkdown形式に変換してください。

【元のテキスト】
本システムは顧客管理を効率化するために開発された。
利用するにはWindows XP以降のOSが必要である。
まず管理者権限でログインし、設定画面から初期設定を行う。
データベースはSQL Server 2005を使用する。
```

**AIの出力例**

```markdown
## 目的

本システムは顧客管理を効率化するために開発された。

## 前提条件

利用するにはWindows XP以降のOSが必要である。

## 手順

まず管理者権限でログインし、設定画面から初期設定を行う。

## 技術要件

データベースはSQL Server 2005を使用する。
```

このように、AIは文脈から各段落の役割を推定し、適切な見出しを付与できます。

#### 9.1.2 領域2：章構造の自動推定と階層化

構造情報が失われた文書（例：PDFから抽出したテキスト、印刷物をOCRで読み取ったテキスト）でも、AIは文章の流れや言語パターンから、章や節の境界を推定できます。

**実践例：構造が失われたテキストの復元**

```markdown
【AIへのプロンプト】
以下のテキストは、構造情報が失われた古い設計書です。
内容を分析し、適切な章・節の階層構造を推定して、
Markdown形式で再構成してください。

【元のテキスト】
システム概要
本システムは在庫管理を目的とする。
背景
従来は手作業で在庫を管理していたが、ミスが多発していた。
目的
在庫の正確な把握と、発注の自動化を実現する。
機能要件
在庫照会機能
現在の在庫数を一覧表示する。
発注機能
在庫が一定数を下回ると自動的に発注する。
```

**AIの出力例**

```markdown
# システム概要

本システムは在庫管理を目的とする。

## 背景

従来は手作業で在庫を管理していたが、ミスが多発していた。

## 目的

在庫の正確な把握と、発注の自動化を実現する。

# 機能要件

## 在庫照会機能

現在の在庫数を一覧表示する。

## 発注機能

在庫が一定数を下回ると自動的に発注する。
```

AIは、「システム概要」と「機能要件」が同じレベルの大見出し（`#`）であり、「背景」「目的」「在庫照会機能」「発注機能」がそれぞれの配下の中見出し（`##`）であることを推定できます。

#### 9.1.3 領域3：用語の統一と表記ゆれの修正

古い文書では、同じ概念を異なる言葉で表現していることがあります（例：「ユーザー」と「利用者」、「データベース」と「DB」）。AIはこれらの表記ゆれを検出し、統一案を提案できます。

**実践例：用語統一のプロンプト**

```markdown
以下の文書内で、同じ概念を指していると思われる異なる表現を
検出し、統一案を提案してください。

【元のテキスト】
ユーザーはログイン画面から認証を行う。
利用者が正しいパスワードを入力すると、システムにアクセスできる。
DBには顧客情報が格納されている。
データベースのバックアップは毎日実施する。
```

**AIの出力例**

```markdown
【検出された表記ゆれ】
1. 「ユーザー」と「利用者」
2. 「DB」と「データベース」

【統一案】
1. → 「ユーザー」に統一（より一般的な表現）
2. → 「データベース」に統一（正式名称を使用）

【修正後のテキスト】
ユーザーはログイン画面から認証を行う。
ユーザーが正しいパスワードを入力すると、システムにアクセスできる。
データベースには顧客情報が格納されている。
データベースのバックアップは毎日実施する。
```

### 9.2 AI変換支援の実践的ワークフロー

AIを活用した変換支援は、段階的なワークフローで進めることで効果を発揮します。以下、実践的な手順を詳しく解説します。

#### 9.2.1 ステップ1：テキスト抽出

まず、一太郎ビューアやPandocなどを使って、元の文書からテキストを抽出します。この段階では、構造情報が失われていても構いません。

#### 9.2.2 ステップ2：AIによる初期変換

抽出したテキストをAIに渡し、Markdown形式への初期変換を依頼します。

**プロンプトのテンプレート**

```markdown
以下のテキストは、[文書の種類：例「設計書」「仕様書」「マニュアル」]から
抽出したものです。このテキストを分析し、以下の要件に従って
Markdown形式に変換してください。

【要件】
1. 章・節の階層構造を推定し、適切な見出しレベル（#, ##, ###）を付与する
2. 箇条書きは`-`または`1.`で表現する
3. 表がある場合はMarkdownのテーブル形式に変換する
4. 専門用語や重要なキーワードは太字（**...**）で強調する
5. 曖昧な表現や不明瞭な箇所があれば、コメントで指摘する

【元のテキスト】
[ここに抽出したテキストを貼り付け]
```

#### 9.2.3 ステップ3：人間によるレビューと調整

AIが生成したMarkdownを、人間が確認します。この段階で以下をチェックします。

- 章・節の階層が論理的に正しいか
- 専門用語の使い方が適切か
- 元の文書の意図が正しく反映されているか
- AIが指摘した曖昧な箇所を明確化する

#### 9.2.4 ステップ4：反復的な改善

必要に応じて、AIに追加の指示を出し、変換を改善します。

**追加指示の例**

```markdown
先ほど変換してもらったMarkdownについて、以下の修正をお願いします：

1. 「システム概要」の章を、「背景」「目的」「対象ユーザー」の3つの節に分割してください
2. 「機能要件」の章に、各機能の「受入基準」を追加してください
3. 用語「ユーザー」を「利用者」に統一してください
```

### 9.3 AIの限界と人間の役割

AIは強力な支援ツールですが、万能ではありません。以下のような限界があります。

**限界1：ドメイン知識の欠如**

AIは、その業務特有の専門知識や背景を持っていません。例えば、「顧客」と「利用者」が実は異なる概念を指している場合、AIはそれを区別できない可能性があります。

**限界2：文脈の誤解**

特に古い文書では、当時の技術的制約や組織の状況を前提とした記述があります。AIはこれらの文脈を正しく理解できないことがあります。

**限界3：図表の解釈**

図面やフローチャートの意味を、AIが完全に理解することは困難です。これらは人間が別途処理する必要があります。

**人間の役割：最終的な判断者**

したがって、AIは「提案者」であり、人間は「最終的な判断者」です。本書で述べた「ヒューマン・イン・ザ・ループ」の「書く→読む」のサンドイッチ構造に従い、**人間が変換すべき文書や要件を明確に書く**ことから始まり、**AIが生成した変換結果を、以下の観点で必ず人間が確認する（読む）**ことで終わります。

- **正確性**：元の文書の意図が正しく反映されているか
- **完全性**：重要な情報が欠落していないか
- **一貫性**：用語の使い方や表現が統一されているか
- **明確性**：将来のメンバーが読んでも理解できるか

これは、本書で繰り返し述べてきた **「Human in the loop」** の原則そのものです。

## 10. 判断基準（そのまま残すか変換するか）

ここまで、OASYSや一太郎の文書をMarkdownに変換する方法を見てきました。しかし、重要な問いがあります：**すべての文書を変換すべきなのでしょうか？**

答えは「No」です。中には、**「そのまま残す」ことが最善の選択である文書**も存在します。

### 10.1 なぜ「そのまま残す」選択肢が必要なのか

文書には、大きく分けて二つの役割があります。

1. **現行の業務を支える「生きた仕様」**：日常的に参照され、更新され、意思決定の基準となる文書
2. **過去の記録としての「歴史資料」**：参照頻度は低いが、法的・記録的価値を持つ文書

前者はMarkdown化して「生きたドキュメント」として管理すべきですが、後者は必ずしもMarkdown化する必要はありません。むしろ、**当時の形式のまま保存する**ことに価値がある場合もあります。

### 10.2 判断基準1：参照頻度と更新頻度

最も重要な判断基準は、**「この文書は今後も参照・更新されるか」**です。

#### 10.2.1 高頻度で参照・更新される文書 → Markdown化を推奨

- 現行システムの仕様書
- 運用手順書
- トラブルシューティングガイド
- API仕様書
- 開発ガイドライン

これらは、チームが日常的に参照し、システムの変更に伴って更新される文書です。Markdown化してGitで管理することで、変更履歴の追跡、複数人での編集、AIとの連携が可能になります。

#### 10.2.2 低頻度でしか参照されない文書 → PDF化してアーカイブ

- 過去のプロジェクトの最終報告書
- 既に廃止されたシステムの設計書
- 契約書や申請書類（法的保管が必要なもの）
- 当時の意思決定の記録

これらは、「何かあったときに参照する」程度の文書です。Markdown化するコストに見合う価値がないため、PDF化して読み取り専用でアーカイブする方が現実的です。

### 10.3 判断基準2：法的・記録的価値

次の判断基準は、**「この文書に法的・記録的価値があるか」**です。

#### 10.3.1 法的価値がある文書 → 原本性を保持してアーカイブ

- 契約書
- 監査資料
- 申請書類
- 承認記録

これらの文書は、**改ざんされていないことを証明できる形式**で保存する必要があります。Markdown化すると、元の形式が失われるため、PDF化（できれば電子署名付き）してアーカイブします。

#### 10.3.2 歴史的価値がある文書 → 当時の形式のまま保存

- 初期の設計思想を示す文書
- アーキテクチャの変遷を示す資料
- 当時の技術的制約を記録した文書

これらは、「なぜそのような設計になったのか」を後から理解するための貴重な資料です。当時の形式（OASYS、一太郎）のまま保存し、必要に応じてビューアで閲覧できる状態にしておきます。

### 10.4 判断基準3：変換コストと期待効果

最後の判断基準は、**「変換にかかるコストと、得られる効果のバランス」**です。

#### 10.4.1 変換コストが低く、効果が高い → 優先的にMarkdown化

- テキスト中心の文書
- 構造がシンプルな文書
- 図表が少ない文書

これらは、PandocやAIを使えば比較的容易にMarkdown化できます。得られる効果（検索性、編集性、AIとの連携）も大きいため、優先的に変換します。

#### 10.4.2 変換コストが高く、効果が低い → PDF化でアーカイブ

- 複雑なレイアウトの文書
- 図表が多い文書
- 特殊なフォントや装飾が重要な文書

これらをMarkdown化するには、図表の個別抽出、レイアウトの再構成など、多大な手作業が必要です。参照頻度が低い場合、このコストに見合う効果は得られません。

### 10.6 判断の実例

**例1：現行システムの運用手順書（一太郎形式）**
- 参照頻度：週に数回
- 更新頻度：月に1回程度
- 法的価値：なし
- 変換コスト：低（テキスト中心）

→ **判断：Markdown化してGit管理**

**例2：10年前のプロジェクトの契約書（OASYS形式）**
- 参照頻度：年に1回あるかないか
- 更新頻度：なし（完了済み）
- 法的価値：あり（保管義務）
- 変換コスト：中

→ **判断：PDF化してアーカイブ（原本性を保持）**

**例3：初期の設計思想を記録した文書（一太郎形式）**
- 参照頻度：低（新メンバー向けの参考資料）
- 更新頻度：なし
- 法的価値：なし
- 歴史的価値：あり
- 変換コスト：中

→ **判断：PDF化してアーカイブ、または一太郎形式のまま保存**

**例4：現行システムのAPI仕様書（一太郎形式）**
- 参照頻度：毎日
- 更新頻度：週に数回
- 法的価値：なし
- 変換コスト：中（表が多い）

→ **判断：Markdown化してGit管理（コストをかけても価値がある）**

### 10.7 「そのまま残す」場合の管理方法

文書をそのまま残す場合でも、適切な管理が必要です。

#### 10.7.1 管理方法1：アーカイブリポジトリの作成

Git管理はしないが、整理された場所に保管します。

```
archive/
├── contracts/           # 契約書
│   ├── 2015-project-A.pdf
│   └── 2018-project-B.pdf
├── historical/          # 歴史的資料
│   ├── initial-design-1995.jtd  # 一太郎形式
│   └── architecture-evolution.pdf
└── README.md            # アーカイブの説明
```

#### 10.7.2 管理方法2：メタデータの記録

各文書について、以下の情報を記録します。

```markdown
# アーカイブ文書一覧

- **期間**：2015年4月〜2016年3月

- **保管期限**：2026年3月まで

## 歴史的資料

### initial-design-1995.jtd
- **文書名**：初期設計思想
- **作成日**：1995年10月
- **保管理由**：アーキテクチャの変遷を理解するための参考資料
- **参照方法**：一太郎ビューアで閲覧
```

## 11. よくある問題と解決策（トラブルシューティング）

### 11.1 一太郎ビューアで開けない

**症状**：一太郎ファイルを開こうとすると、エラーが表示される

**対処法**：
1. 一太郎ビューアのバージョンを確認（Ver.2以降のファイルに対応）
2. ファイルが破損していないか確認
3. 古いバージョンの一太郎ファイルの場合、文字化けや書式崩れのリスクがあることを理解する

### 11.2 OASYS Viewerが入手できない

**症状**：OASYS Viewerが入手できない、または古いOSでしか動作しない

**対処法**：
1. 組織内に過去に入手したOASYS Viewerが残っていないか確認
2. Windows XP環境（仮想環境）を用意できるか検討（セキュリティリスクに注意）
3. 印刷物をスキャンして画像化する方法を検討
4. 専門業者への依頼を検討

### 11.3 PDF化した文書の検索性が低い

**症状**：PDF化した文書で全文検索ができない、または精度が低い

**対処法**：
1. OCR（光学文字認識）機能付きスキャナーを使用
2. PDF編集ソフト（Adobe Acrobat、PDFelementなど）でOCR処理を実行
3. 検索精度には限界があることを理解し、重要な文書はMarkdown化を検討

### 11.4 構造情報が失われた

**症状**：PDF化や画像化により、見出し構造や段落構造が失われた

**対処法**：
1. AI変換支援（9節）を活用して構造を推定
2. 元のファイルが残っている場合は、Word経由の変換（8節）を検討
3. 手作業での構造復元が必要な場合、優先度の高い文書から着手

### 11.5 図面の関係性が失われた

**症状**：図面を画像化した結果、要素間の関係性が失われた

**対処法**：
1. 可能であれば、元のファイルから図形を個別に抽出（5節参照）
2. SVG形式で保存し、テキスト情報を保持
3. Markdown内に図面の説明文を記述し、意味を補完

### 11.6 文字化けが発生する

**症状**：変換後のMarkdownで日本語が文字化けする

**対処法**：
1. ファイルのエンコーディングをUTF-8に統一
2. VS CodeやCursorで「エンコーディング付きで再度開く」→ 正常表示 → 「UTF-8で保存」
3. 元のファイルがShift-JISなどの場合、変換時にエンコーディングを指定

### 11.7 見出し階層が崩れる

**症状**：変換後のMarkdownで見出しレベルが飛んでいる（`#`の次にいきなり`###`になっている）

**対処法**：
1. 変換前に元の文書で見出しスタイルを確認
2. 変換後に手作業で階層を修正
3. 正規表現を使って一括修正（VS Codeの置換機能など）

例：`1.1 概要` の行を見出しにする（レベル2）
- 検索（正規表現）：`^([0-9]+\.[0-9]+.*)`
- 置換：`## $1`

```markdown
以下のテキストは、[文書の種類]から抽出したものです。
このテキストを分析し、以下の要件に従ってMarkdown形式に変換してください。

【要件】
1. 章・節の階層構造を推定し、適切な見出しレベル（#, ##, ###）を付与する

4. 専門用語や重要なキーワードは太字（**...**）で強調する

[ここに抽出したテキストを貼り付け]
```

### 12.3 変換ワークフロー（短縮版）

1. 変換対象を選ぶ（更新される文書から）  
2. ビューワーでPDF化・画像化（文書を失わない）  
3. 構造抽出（図面・段落・スタイル）  
4. Word経由の変換（必要に応じて）  
5. AI変換支援（構造推定・タグ付け）  
6. 人間によるレビューと調整  
7. Git管理＋PRレビュー  
8. 品質は運用で磨く

### 12.4 判断フローチャート（簡易版）

```
参照・更新される？
  → Yes → Markdown化してGit管理
  → No → 法的・歴史的価値がある？
    → Yes → PDF化してアーカイブ
    → No → 廃棄検討
```
