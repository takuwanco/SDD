## 第1章：仕様駆動開発の全体像

### 1.1 Markdownで仕様を書く基本プロンプト

**目的**: Markdown形式で仕様を書き始める

```
[プロジェクト名]のREADME.mdを作成してください。

含めるべき項目：
1. プロジェクト概要：何を作るのか（What）
2. 目的：なぜ作るのか（Why）
3. 主要機能：どんな機能があるか
4. 技術スタック：使用する技術
5. Getting Started：始め方

形式：GitHub Flavored Markdown
トーン：技術者にも非技術者にも理解できる平易な言葉
```

### 1.2 4つの原則を仕様に反映させるプロンプト

**目的**: 仕様駆動開発の4つの原則を意識した仕様を書く

```
[機能名]の仕様をMarkdown形式で書いてください。
以下の4つの原則を反映してください：

1. 生きたドキュメント：変更履歴を追跡できるように、変更の理由を記載
2. 信頼できる唯一の情報源：この仕様が唯一の正式な情報源であることを明記
3. 変更と反復が前提：将来の変更を想定した柔軟な構造
4. AIでコストを抑える：AIが理解しやすい明確な構造

ファイル名：docs/spec/[機能名].md
```

## 第2章：30分で実感する仕様駆動開発

### 2.1 30分でREADME.mdを作成するプロンプト

**目的**: 最初の30分で仕様を書き始める

```
私は[プロジェクト名]を開発しています。
30分でREADME.mdの初期版を作成したいです。

プロジェクトの概要：
[プロジェクトの簡単な説明を1-2行で]

以下を含めてください：
- プロジェクトタイトル
- What（何を作るか）
- Why（なぜ作るか）
- 主要機能（3つ程度）
- 次のステップ（TODOリスト）

形式：Markdown
長さ：200-300語程度
```

### 2.2 仕様からコードを生成するプロンプト

**目的**: README.mdの仕様からコードを生成する

```
README.mdに記載された仕様を読んで、[機能名]のコードを生成してください。

仕様ファイル：README.md
セクション：[該当するセクション名]

生成してほしいもの：
- ファイル名：[ファイル名]
- 言語：[プログラミング言語]
- フレームワーク：[フレームワーク名]

仕様との整合性を保つため、コードにコメントで仕様へのリンクを含めてください。
```

## 第3章：Cursorで仕様を記述・活用する

### 3.1 AIチャットパネルで仕様を作成するプロンプト

**目的**: Cursorのチャット機能で仕様を対話的に作成する

```
[機能名]の仕様を作成したいです。
段階的に質問してください。

質問してほしい項目：
1. この機能の目的は何ですか？（Why）
2. 何を実現しますか？（What）
3. 受入基準は何ですか？（どうなれば完成か）
4. 制約条件はありますか？
5. 関連する既存機能はありますか？

各質問に答えた後、最終的にMarkdown形式の仕様書を生成してください。
```

### 3.2 Composerで複数ファイルを同時に更新するプロンプト

**目的**: 仕様とコードを同時に更新する

```
以下の変更を行ってください：

1. docs/spec/[機能名].md の仕様を更新
   - [変更内容を記載]

2. src/[ファイル名] のコードを仕様に合わせて更新
   - [変更内容を記載]

3. tests/[テストファイル名] のテストを追加/更新
   - [変更内容を記載]

すべての変更が整合性を保つようにしてください。
コードには仕様へのリンクをコメントで含めてください。
```

### 3.3 Cmd+Kでコードの一部を修正するプロンプト

**目的**: コードの特定部分を仕様に合わせて修正する

```
この関数を以下の仕様に合わせて修正してください：

仕様：docs/spec/[機能名].md#[セクション名]
要件：[具体的な要件]

修正後、仕様へのリンクをコメントで追加してください。
```

### 3.4 コードから仕様を逆生成するプロンプト

**目的**: 既存のコードから仕様を抽出する

```
以下のコードを読んで、仕様書（Markdown）を生成してください。

対象コード：[ファイル名]

仕様書に含めるべき内容：
- このコードが「何をするか」（What）
- なぜこの実装が必要か（Why）を推測
- 受入基準：このコードがパスすべきテスト項目
- 既存の仕様（docs/spec/*.md）との整合性を確認

出力先：docs/spec/[機能名].md
形式：GitHub Flavored Markdown
```

### 3.5 Cursorの設定ファイル（.cursorrules）用プロンプト

**本文での説明**: 第3章 3.9節「Cursorの設定をカスタマイズする（.cursorrules）」で、基本的な`.cursorrules`の使い方を説明しています。このセクションでは、より詳細な設定例を提供します。

#### .cursorrulesファイルの作成プロンプト

**目的**: 仕様駆動開発を支援する`.cursorrules`ファイルを作成する

**使い方**: プロジェクトのルートディレクトリに`.cursorrules`ファイルを作成し、以下の内容を記述します。

##### 基本的な.cursorrules（小規模チーム向け）

```
あなたは仕様駆動開発を実践するプロジェクトのアシスタントです。

## 基本ルール

1. コードを生成する際は、必ず関連する仕様ファイル（docs/spec/*.md または README.md）を参照する
2. 仕様が不明確な場合は、コードを生成する前に質問する
3. 生成したコードには、仕様へのリンクをコメントで含める
   - 例（Python）: # 仕様: docs/spec/authentication.md#ログイン処理
   - 例（JavaScript）: // 仕様: docs/spec/authentication.md#ログイン処理
4. 仕様ファイルを更新する際は、Why（なぜ）とWhat（何を）を明確にする
5. 実装の詳細（How）は仕様に書かない

## 仕様記述のルール

- 見出しは明確で具体的にする
- 曖昧な表現（「適切に」「必要に応じて」など）は避ける
- 受入基準は測定可能にする
- 変更理由は必ず記録する
```

##### 充実した.cursorrules（中規模〜大規模組織向け）

```
あなたは仕様駆動開発を実践するプロジェクトのアシスタントです。

## プロジェクト構造

- 仕様ファイル: docs/spec/*.md
- コード: src/
- テスト: tests/
- プロジェクト憲章: Constitution.md（または README.md内）

## 仕様駆動開発の4つの原則

1. **生きたドキュメント**: 仕様は常に最新の状態を保つ
2. **信頼できる唯一の情報源**: 仕様は一箇所にまとめ、分散させない
3. **変更と反復が前提**: 将来の変更を想定した柔軟な構造
4. **AIでコストを抑える**: AIが理解しやすい明確な構造

## コード生成時のルール

1. **仕様ファイルを必ず参照**
   - コードを生成する前に、関連する仕様ファイル（docs/spec/*.md）を読む
   - 仕様が存在しない場合は、ユーザーに確認する

2. **仕様へのリンクを含める**
   - 生成したコードには、必ず仕様へのリンクをコメントで含める
   - 形式: `# 仕様: docs/spec/[ファイル名].md#[セクション名]`

3. **不明点は質問する**
   - 仕様が曖昧な場合は、コードを生成する前に質問する
   - 「適切に」「必要に応じて」などの表現があれば、具体化を求める

## 仕様記述時のルール

1. **Why（なぜ）を明確にする**
   - 「なぜこの機能が必要か」を必ず記述する
   - ビジネス的な背景や目的を含める

2. **What（何を）を具体的にする**
   - 「何を実現するか」を明確に記述する
   - 曖昧な表現を避ける

3. **受入基準を測定可能にする**
   - 「どうなれば完成か」を明確にする
   - テスト可能な形で記述する

4. **実装の詳細（How）は書かない**
   - 「どう実装するか」は仕様に含めない
   - コードで表現する

## Pull Request作成時のルール

1. **仕様の変更を含める**
   - コードの変更に伴い、仕様ファイルも更新する
   - 仕様の変更理由を記録する

2. **トレーサビリティを保つ**
   - PR説明文に関連する仕様へのリンクを含める
   - Issue番号があれば記載する

## やってはいけないこと

1. 仕様なしにコードを書かない
2. 仕様とコードを分断させない（リンクを必ず含める）
3. 実装の詳細（How）を仕様に書かない
4. 曖昧な表現（「適切に」「必要に応じて」）を使わない
5. 受入基準を曖昧にしない

## 出力形式

- 仕様ファイル: GitHub Flavored Markdown
- コメント: 仕様へのリンクを必ず含める
- コミットメッセージ: Conventional Commits形式（feat:, fix:, docs:など）
```

#### .cursorrulesを活用したプロンプト例

**目的**: .cursorrulesの設定を活かして効率的に開発する

##### 例1: 仕様から実装を生成

## 第5章：最初の1週間で仕様駆動開発を始める方法

### 5.1 プロジェクト名から仕様を書き始めるプロンプト

**目的**: 「始められない」を突破する

```
プロジェクト名：[プロジェクト名]

このプロジェクト名から、README.mdの最初の1段落を書いてください。

含めるべき内容：
- プロジェクト名の由来
- 何を作るか（1文で）
- なぜ作るか（1文で）

長さ：50-100語
トーン：わかりやすく、前向きな表現
```

### 5.2 議事録を仕様に変換するプロンプト

**目的**: 「続かない」を突破する

```
以下の議事録から、仕様として記録すべき内容を抽出し、
README.mdに追加する形式で出力してください。

議事録：
[議事録の内容]

抽出してほしい内容：
- 決定事項（What）
- 決定理由（Why）
- 次のアクション（TODO）
- 関連するIssue番号（あれば）

出力形式：Markdown
追加先セクション：## 変更履歴
```

### 5.3 小さな成功を可視化するプロンプト

**目的**: 「広がらない」を突破する

```
今週のREADME.mdの変更点をまとめて、
チームに共有するレポートを作成してください。

含めるべき内容：
1. 今週追加された内容（箇条書き）
2. 改善された点（Before/After）
3. 次週の目標（3つ程度）
4. 数値化できる成果（例：仕様の行数、Pull Request数）

形式：Markdown
トーン：前向きで、成果を強調する表現
```

### 5.4 AIから質問してもらう壁打ちプロンプト（**重要**）

**目的**: 仕様の品質を大幅に向上させる

**このプロンプトの重要性**:
AIに答えを求めるのではなく、**AIから質問してもらい、人間が答える**という発想の転換が、仕様の品質を大きく向上させます。第5章で詳しく説明されている最も効果的な手法です。

```
あなたは上級SEです。このREADME.mdを読んで、不明な点を5つ質問してください。

# 仕様書レビュー指示

## インプット情報
### README.md
（CursorのAIチャットパネルは、開いているREADME.mdを自動的に読み取ります）

## アウトプット情報
### 出力形式
- 不明な点を5つ、箇条書きで出力してください
- それぞれの質問に対して、なぜ不明なのかを1行で説明してください

## 注意事項
- 技術的な詳細よりも、仕様の明確さに焦点を当ててください
- 質問は、実装に影響する重要な点に絞ってください
```

**期待されるAIからの質問例**:
- 「想定ユーザーの『倉庫管理者』は、1日あたりどのくらいの入出庫処理を想定していますか？」
- 「在庫アラート機能は、どのような条件でアラートを発するのですか？」
- 「リアルタイム表示の更新頻度は、どのくらいを想定していますか？」

**使い方**:
1. CursorのAIチャットパネルを開く
2. README.mdを開いた状態で、上記のプロンプトを入力
3. AIからの質問に答え、README.mdに追加する
4. 必要に応じて「この質問に対する答えをREADME.mdに追加してください」と指示

### 5.5 プロンプトの構造化（ロール・指示・出力形式・制約）

**目的**: AIからの回答の品質を向上させる

```

[タスクの説明]

## インプット情報
[AIに読んでほしい情報]

## アウトプット情報
### 出力形式
```

**目的**: よくある失敗パターンを事前に検出する

```

チェック項目：

## 仕様駆動開発の原則違反
- [ ] 仕様が「生きたドキュメント」ではなく、更新されていない
- [ ] 信頼できる唯一の情報源ではなく、情報が分散している
- [ ] 変更と反復が前提になっていない（柔軟性がない）
- [ ] AIが理解しにくい構造になっている

## よくある失敗パターン
- [ ] 過剰形式主義：形式的な記述が多く、本質的な情報が少ない
- [ ] 属人性：特定の人に依存した仕様になっている
- [ ] 仕様の分断：情報が複数の場所に散在している
- [ ] 実装の詳細（How）を仕様に書いている

## セキュリティリスク
- [ ] 機密情報（パスワード、APIキーなど）が含まれている
- [ ] 個人情報が含まれている
- [ ] セキュリティ要件が曖昧で、実装が不明確

## コミュニケーション不足
- [ ] Whyが不明確（なぜ必要かが書かれていない）
- [ ] 受入基準が曖昧（どうなれば完成かが不明）
- [ ] 関連する仕様へのリンクがない
```

```markdown
- **問題点**: 「ログイン機能を実装する」とだけ書かれており、なぜ必要かが不明
- **改善案**: 「ユーザーの個人情報を保護するため、認証機能が必要」と追記

### ❌ 問題2: 機密情報が含まれている
- **該当箇所**: config.md の「APIキー」セクション
- **問題点**: APIキーが平文で記載されている
- **改善案**: 環境変数として管理し、.envファイルは.gitignoreに追加

## チェック結果
- ✅ 合格項目: 8/12
- ⚠️ 要改善項目: 4/12
```

```
   - 仕様とコードが完全に分離され、リンクがない
   - 仕様がコードから参照されていない

2. **仕様の形骸化**

   - 受入基準が測定不可能

5. **承認プロセスの形骸化**
   - Pull Requestが「LGTM」だけで承認されている
   - レビューコメントがほとんどない
```

## プロンプトのカスタマイズ方法

### 1. チーム規模に合わせたカスタマイズ

**小規模チーム（3-5人）向け**
- プロンプトをシンプルにする
- 「承認者：1名」など、承認プロセスを簡略化する表現を追加

**中規模プロジェクト（6-30人）向け**
- 「関連チーム」「担当者」などの項目を追加
- 「CODEOWNERSに基づいて適切なレビュアーを割り当てる」などの指示を追加

**大規模組織（50人以上）向け**
- 「稟議書番号」「監査証跡」などの項目を追加
- 「階層的な承認フロー」を意識した表現を追加

### 2. プログラミング言語に合わせたカスタマイズ

プロンプト内の`[プログラミング言語]`を実際の言語に置き換えます：

- Python
- JavaScript/TypeScript
- Java
- Go
- Rust
- など

### 3. 組織の文化に合わせたカスタマイズ

**フォーマルな組織**
- 「です・ます調」で丁寧な表現を指定
- 「稟議」「承認」などの正式な用語を使用

**カジュアルな組織**
- 「だ・である調」で簡潔な表現を指定
- 「レビュー」「OK」などのカジュアルな用語を使用

## プロンプトの効果的な使い方

### 1. 段階的に詳細化する

最初は簡単なプロンプトから始め、必要に応じて詳細化します。

**例：段階1（シンプル）**
```
README.mdを作成してください。
```

**例：段階2（詳細）**
```
[プロジェクト名]のREADME.mdを作成してください。
含めるべき項目：プロジェクト概要、目的、主要機能
```

**例：段階3（カスタマイズ）**
```
[プロジェクト名]のREADME.mdを作成してください。
含めるべき項目：プロジェクト概要、目的、主要機能、技術スタック
形式：GitHub Flavored Markdown
トーン：非技術者にも理解できる平易な言葉
```

### 2. コンテキストを明示する

AIが適切な判断をするために、コンテキストを明示します。

```
私は[役割]です。
[プロジェクト名]で[機能名]を開発しています。
既存の仕様：docs/spec/*.md
既存のコード：src/*.py

[プロンプトの内容]
```

### 3. 出力形式を明確にする

期待する出力形式を具体的に指定します。

```
出力形式：GitHub Flavored Markdown
ファイル名：docs/spec/[機能名].md
セクション構造：
## Why（なぜ）
## What（何を）
## 受入基準
## 制約条件
```

### 4. フィードバックループを活用する

AIの出力をレビューし、改善を繰り返します。

```
先ほど生成した仕様を見直してください。
以下の点を改善してください：
1. [改善点1]
2. [改善点2]
3. [改善点3]
```

## まとめ

このプロンプト集は、本書で紹介されているAI活用シーンに対応した実践的なプロンプトを収録しています。

**活用のポイント**:

1. **そのまま使わず、カスタマイズする**: プロンプトは出発点です。自分のプロジェクトに合わせてカスタマイズしてください
2. **段階的に詳細化する**: 最初は簡単なプロンプトから始め、必要に応じて詳細化します
3. **コンテキストを明示する**: AIが適切な判断をするために、コンテキストを明示します
4. **フィードバックループを回す**: AIの出力をレビューし、改善を繰り返します
5. **チームで共有する**: 効果的なプロンプトをチームで共有し、改善を重ねます

**次のステップ**:

- このプロンプト集をコピーして、自分のチームに合わせてカスタマイズ
- 新しい効果的なプロンプトを見つけたら、この集に追加
- チームメンバーと共有し、フィードバックを得る
